# 05 工具链

## 主流工具横向对比

市面上和 Harness 相关的工具大致分两类：一类是 AI 编程工具（直接帮你写代码的），一类是 Harness 框架（帮你搭管控脚手架的）。两类都得了解，因为它们解决的不是同一个问题。

### AI 编程工具

| 工具 | 定位 | 内置 Harness 能力 | 适合场景 |
|------|------|-------------------|----------|
| **Claude Code** | CLI 形式的 AI 编程助手 | 任务分解、自动重试、代码审查、进度追踪 | 需要深度控制工作流、喜欢命令行 |
| **Cursor Agent** | AI IDE，VS Code 内核 | 多步任务规划、错误拦截、上下文管理 | 日常开发、想要 IDE 体验 |
| **Roo Code** | VS Code 插件（原 Cline） | 完整的任务队列、工具调用管控、自动检查 | 想在 VS Code 里用接近 Claude Code 的体验 |
| **Codex** | OpenAI 的代码生成 API | 基础任务执行，Harness 能力较弱 | 嵌入到自己的工作流，自己加管控层 |

### Harness 框架

| 框架 | 定位 | 成熟度 | 适合场景 |
|------|------|--------|----------|
| **LangGraph** | 状态机风格的 Agent 工作流框架 | 高 | 复杂多步工作流，需要精确控制状态转移 |
| **AutoGPT** | 自动化任务执行，自主 Agent | 中 | 快速实验，对可靠性要求不极高 |
| **CrewAI** | 多 Agent 协作框架 | 中 | 需要多个 AI 角色分工配合的场景 |
| **OpenAI Agents SDK** | OpenAI 官方 Agent 基础设施 | 高（但生态较新） | 重度依赖 OpenAI 模型的项目 |

## 主流工具已经内置了 Harness 的思想

这是一个值得注意的现实：你现在打开 Cursor Agent 或者 Claude Code，它们做的事情和 Harness Engineering 描述的核心循环高度重合。

以 Roo Code 为例，你说「帮我重构登录模块」，它背后在做的事是：

```
1. 拆解任务：分析登录模块涉及哪些文件
2. 制定计划：列出修改步骤
3. 逐步执行：一个文件一个文件地改
4. 自动检查：每步改完跑 linter
5. 错误处理：遇到测试失败自动分析并重试
6. 进度记录：在对话里保持任务状态
```

这就是 Harness。只是它被包装成了对话界面，你感知不到它在背后做这些事。

换句话说，如果你在用这些主流工具，你已经在用 Harness 了。Harness Engineering 这个概念的价值，更多在于帮你理解「为什么要这样设计工作流」，以及在这些工具不够用的时候「如何自己搭一套更定制化的管控层」。

---

## 但工具再完善，模型撑不住这个事实不变

主流工具已经把 Harness 的思想落地得很好了，任务分层、代码审查、进度追踪，这些机制都在运转。

但有一件事是工具解决不了的：**模型本身在处理复杂逻辑时的长程推理能力不足**。

工具可以把一个大任务切成 50 个小任务，让 AI 一步一步做。但每一步之间的「接口约定是否一致」「状态流转是否正确」「这个决策和 30 步前的决策有没有矛盾」——这些需要全局视角的判断，工具帮不了你，还是要靠模型自己维持。而这恰好是现在模型最容易在长任务里崩掉的地方。

所以不管你用的是哪个工具，都要保持清醒：工具是放大器，不是救世主。模型能力到了天花板，再好的工具也只是延缓衰退，不能突破上限。

## Vercel 的案例：减法反而提升了成功率

这是 Harness Engineering 里一个很有说服力的真实案例。

Vercel 团队在给 AI 配工具的时候，最初的思路是「尽量给 AI 准备好专用工具」——专门的文件读写工具、专门的搜索工具、专门的代码分析工具，每一类任务都有对应的定制化工具。

结果发现这样做成功率反而上不去，稳定在 80% 左右。AI 在多个专用工具之间切换，经常选错工具，或者一个工具出了问题整个流程就卡住了。

后来他们反过来做了一件事：**去掉 80% 的手工专用工具，让 AI 直接用原生通用工具**——`npm`、`grep`、`curl`，就是开发者日常用的那些命令行工具。

成功率从 80% 升到了 100%。

原因其实不难理解。通用工具有几个优势：
- AI 训练数据里见过大量这些工具的使用案例，理解更准确
- 通用工具的行为稳定可预测，不会因为封装层的问题引入意外
- AI 遇到错误时更容易自己诊断，因为错误信息是标准的、它认识的

这个案例背后有一个更深的道理，在 AI 领域被称为「苦涩的教训」。

---

## 苦涩的教训（The Bitter Lesson）

Richard Sutton 在 2019 年写过一篇文章，标题就叫「The Bitter Lesson」。核心观点是：在 AI 发展史上，**利用通用计算能力（算力 + 数据）的方法，长期来看总是打败依赖人类专业知识的方法**。

每一次 AI 研究者精心设计了一套基于领域知识的系统，过几年都会被「算力更大 + 数据更多 + 更通用的架构」给干掉。象棋 AI 里的开局库、围棋 AI 里的人类棋谱、语音识别里的声学模型……无一例外。

这个教训放到 Harness Engineering 里同样适用。手工精心设计的专用工具、精雕细琢的 prompt 模板、高度定制化的工作流——这些在短期内有效，但随着模型能力提升，会逐渐被更通用的做法取代。

Vercel 的案例就是一个缩影：把手工知识（专用工具）去掉，换成通用计算（原生命令），效果反而更好。

这不是说精心设计没有价值，而是说设计的方向要对：**让 AI 用它本来就懂的通用工具，而不是强迫它学习你专门封装的那一套**。

对 Harness 的设计也是一样的启示：Harness 的结构越简单、越接近标准工具链，AI 在其中工作的可靠性就越高。过度工程化的 Harness 会成为负担，不是助力。

## 框架安装入口

如果你想快速上手某个工具或框架，以下是各主要工具的安装方式：

```bash
# Python 生态
pip install langgraph              # LangGraph：状态机风格，适合复杂工作流
pip install crewai                 # CrewAI：多 Agent 协作
pip install openai-agents          # OpenAI Agents SDK
pip install langchain              # LangChain：工具链整合

# Node.js 生态
npm install @openai/agents-sdk     # OpenAI 官方 Agents SDK（JS）
npm install langchain              # LangChain JS 版
npm install @langchain/langgraph   # LangGraph JS 版

# Claude Code（CLI 工具，独立安装）
npm install -g @anthropic-ai/claude-code

# Roo Code
# VS Code 扩展市场搜索 "Roo Code" 安装

# Cursor
# 从 cursor.com 下载安装包
```

如果你只是想快速体验 Harness 的核心循环，不需要装任何框架。用 Python 或 Node.js 写 50 行脚本，把任务队列、检查、记忆三样东西串起来，就是一个能跑的 Harness。框架帮你省掉的是胶水代码，但理解原理这件事，框架替代不了。

## 小结

工具链这块的核心结论可以归纳成三句话：

第一，**主流 AI 编程工具已经内置了 Harness 的思想**，任务分层、代码审查、进度追踪都在运转。你在用 Cursor Agent 或 Claude Code 的时候，Harness 已经在背后工作了。

第二，**工具再完善，解决不了模型本身在复杂逻辑里的长程推理能力问题**。这是当前模型的能力边界，工具只能延缓衰退，不能突破上限。

第三，**通用工具优于专用工具**。Vercel 去掉 80% 专用工具后成功率反而从 80% 升到 100%，背后是「苦涩的教训」在发挥作用：AI 用它本来就懂的通用工具，比强迫它学习你专门封装的那一套可靠得多。

设计 Harness 和选择工具链的原则是一样的：越简单、越接近标准、越通用，AI 在其中工作的可靠性就越高。
