# 03 构建方式

## 核心三件套

Harness 的落地不需要什么神秘框架，最朴素的实现就是三样东西：**任务队列、检查器、记忆库**。把这三样组合起来，就已经比绝大多数「写完提示词扔给 AI 随便跑」的方式强一个量级。

| 组件 | 职责 | 类比 |
|------|------|------|
| 任务队列 | 定义 AI 先做什么后做什么，每次只给一个任务 | 工头手里的施工清单 |
| 检查器 | AI 每完成一步就自动验证，不合格直接打回 | 质检员，不是事后审查 |
| 记忆库 | 记录做了什么、做到哪了、出过什么错 | 工程日志，防止 AI 失忆 |

三者缺一不可。光有任务队列没有检查器，AI 跑偏了你不知道；光有检查器没有记忆库，下一步开始时 AI 又忘了上一步做了什么。

## 最简 Harness 脚本（Python）

```python
import subprocess
import json
from pathlib import Path

PROGRESS_FILE = Path("harness/progress.json")

def load_progress():
    if PROGRESS_FILE.exists():
        return json.loads(PROGRESS_FILE.read_text(encoding="utf-8"))
    return {}

def save_progress(task_id, status, notes=""):
    data = load_progress()
    data[task_id] = {"status": status, "notes": notes}
    PROGRESS_FILE.write_text(json.dumps(data, ensure_ascii=False, indent=2), encoding="utf-8")

def passes_linter(filepath):
    result = subprocess.run(["npx", "eslint", filepath], capture_output=True)
    return result.returncode == 0

def run_tests():
    result = subprocess.run(["npm", "test", "--", "--watchAll=false"], capture_output=True)
    return result.returncode == 0

def write_code(task_id, ai_output, filepath):
    """
    Harness 的核心门禁：AI 每次输出都要过这里
    """
    # 写入临时文件先检查
    tmp = Path(f"/tmp/{task_id}.js")
    tmp.write_text(ai_output, encoding="utf-8")

    if not passes_linter(str(tmp)):
        save_progress(task_id, "FAIL", "linter 未通过")
        return False, "代码风格不合规，打回重写"

    if not run_tests():
        save_progress(task_id, "FAIL", "测试未通过")
        return False, "测试失败，打回重写"

    # 全部通过才落地
    Path(filepath).write_text(ai_output, encoding="utf-8")
    save_progress(task_id, "PASS")
    return True, "通过"
```

这就是一个能跑的 Harness 骨架。没有任何魔法，就是「AI 输出 → 检查 → 不过就返回错误信息让 AI 重写 → 过了才保存」这个循环。

## 带任务队列的 JS 版本

```javascript
// harness/runner.js
const { execSync } = require('child_process');
const fs = require('fs');

const TASKS = [
  { id: 'auth',     file: 'src/auth.js',     prompt: '写用户认证模块，JWT方案' },
  { id: 'articles', file: 'src/articles.js',  prompt: '写文章CRUD，RESTful接口' },
  { id: 'comments', file: 'src/comments.js',  prompt: '写评论模块，含嵌套回复' },
];

const MEMORY_FILE = 'harness/memory.json';

function loadMemory() {
  if (fs.existsSync(MEMORY_FILE)) {
    return JSON.parse(fs.readFileSync(MEMORY_FILE, 'utf-8'));
  }
  return {};
}

function saveMemory(taskId, result) {
  const mem = loadMemory();
  mem[taskId] = { ...result, timestamp: new Date().toISOString() };
  fs.writeFileSync(MEMORY_FILE, JSON.stringify(mem, null, 2), 'utf-8');
}

function check(filepath) {
  try {
    execSync(`npx eslint ${filepath}`, { stdio: 'pipe' });
    execSync('npm test -- --watchAll=false', { stdio: 'pipe' });
    return { pass: true };
  } catch (e) {
    return { pass: false, error: e.stderr?.toString() || e.message };
  }
}

async function runTask(task, aiClient) {
  const memory = loadMemory();

  // 已经通过的不重跑
  if (memory[task.id]?.status === 'PASS') {
    console.log(`[SKIP] ${task.id} 已完成`);
    return;
  }

  let attempts = 0;
  const MAX_RETRY = 3;

  while (attempts < MAX_RETRY) {
    attempts++;
    console.log(`[RUN] ${task.id}，第 ${attempts} 次尝试`);

    const code = await aiClient.generate(task.prompt);
    fs.writeFileSync(task.file, code, 'utf-8');

    const result = check(task.file);
    if (result.pass) {
      saveMemory(task.id, { status: 'PASS', attempts });
      console.log(`[PASS] ${task.id}`);
      return;
    }

    console.log(`[FAIL] ${task.id}：${result.error}`);
    saveMemory(task.id, { status: 'FAIL', attempts, error: result.error });

    // 把错误信息塞回 prompt，让 AI 知道哪里错了
    task.prompt = `${task.prompt}\n\n上次尝试失败，错误信息：\n${result.error}\n请修复后重写。`;
  }

  console.error(`[ABORT] ${task.id} 超过最大重试次数`);
}

async function main(aiClient) {
  for (const task of TASKS) {
    await runTask(task, aiClient);
  }
  console.log('全部任务完成');
}

module.exports = { main };
```

## 实际项目目录结构

一个完整的 Harness 化项目长这个样子：

```
my-project/
│
├── .husky/                     # Git 钩子目录（自动触发检查）
│   ├── pre-commit              # 提交前：跑 linter + 类型检查
│   └── pre-push                # 推送前：跑完整测试套件
│
├── harness/                    # Harness 工程核心目录
│   ├── task-queue.js           # 任务队列：定义开发顺序和依赖关系
│   ├── linter-rules.js         # 自定义代码规范检查规则
│   ├── arch-test.js            # 架构约束测试（ArchUnit 风格）
│   ├── progress-tracker.js     # 进度追踪：读写 progress 文件
│   ├── garbage-collector.js    # 定期扫描并清理 AI 废料
│   └── e2e-validator.js        # 端到端验证：Puppeteer 截图验证
│
├── harness/features.json       # 进度文件：所有功能点，默认 FAIL
├── harness/memory.json         # 记忆库：记录每步执行结果
│
├── AGENTS.md                   # AI 智能体索引文档（可理解环境核心）
├── docs/
│   ├── architecture.md         # 系统架构说明
│   ├── api-spec.md             # API 规范
│   └── constraints.md          # 开发约束和禁止事项
│
├── src/                        # 业务代码
├── tests/                      # 测试用例
└── run.js                      # Harness 启动入口
```

`.husky` 和 `harness/` 是 Harness 化项目的两个标志性目录。前者负责 Git 层面的门禁，后者负责 AI 执行层面的管控。

## 进度文件：features.json 的做法

这是 Harness 里一个非常有价值的实践，思路来源于「initializer 智能体」的概念。

项目开始时，先让 AI 在满血状态下（上下文最干净的时候）把整个需求拆解成数百个细粒度功能点，写进 `features.json`，**全部默认设置为 `FAIL`**。这个文件从此成为项目的唯一真相来源。

```json
{
  "auth/register": { "status": "FAIL", "desc": "用户注册接口，含邮箱验证" },
  "auth/login":    { "status": "FAIL", "desc": "JWT 登录，refresh token 机制" },
  "auth/logout":   { "status": "FAIL", "desc": "登出并使 token 失效" },
  "article/create": { "status": "FAIL", "desc": "创建文章，含草稿功能" },
  "article/list":   { "status": "FAIL", "desc": "文章列表，分页+过滤" },
  "article/edit":   { "status": "FAIL", "desc": "编辑文章，版本历史" },
  "comment/create": { "status": "FAIL", "desc": "发表评论，嵌套回复" }
}
```

为什么默认全是 FAIL？这是一种**强制步步为营**的机制。AI 每完成一个功能并通过检查，才能把对应条目从 `FAIL` 改成 `PASS`。

好处很明显：
- AI 无法跳步，不能声称「整个模块搞定了」但实际上有一半没做
- 中断后可以随时续跑，从最后一个 `FAIL` 继续
- 项目进度一眼就看清楚，不需要问 AI「你做到哪了」
- 人工审查时有明确的 checklist，不会漏查

类似的做法是用 `progress.txt`，但 JSON 更适合机器读写，也更容易做进度统计。

```python
# 统计进度
import json
data = json.load(open('harness/features.json', encoding='utf-8'))
total = len(data)
passed = sum(1 for v in data.values() if v['status'] == 'PASS')
print(f"进度：{passed}/{total} ({passed/total*100:.1f}%)")
```

## 端到端验证循环：让 AI 自己截图验证

代码通过 linter 和单元测试只是必要条件，不是充分条件。一个函数签名完全正确、测试全绿，但页面上按钮点了没反应——这种情况太常见了。

解决方法是给 AI 配上 Puppeteer 或 Chrome DevTools Protocol，让它自己打开浏览器、操作页面、截图，然后把截图发给视觉模型做验证。整个循环不需要人参与。

```javascript
// harness/e2e-validator.js
const puppeteer = require('puppeteer');
const path = require('path');

async function validateFeature(featureId, steps) {
  const browser = await puppeteer.launch({ headless: true });
  const page = await browser.newPage();
  await page.setViewport({ width: 1280, height: 800 });

  const screenshots = [];

  for (const step of steps) {
    await page.goto(step.url);

    if (step.action === 'click') {
      await page.click(step.selector);
    } else if (step.action === 'fill') {
      await page.type(step.selector, step.value);
    }

    await page.waitForTimeout(500);

    const screenshotPath = path.join('harness/screenshots', `${featureId}_${step.name}.png`);
    await page.screenshot({ path: screenshotPath, fullPage: false });
    screenshots.push({ step: step.name, path: screenshotPath });
  }

  await browser.close();

  // 把截图路径列表返回给 AI，让视觉模型判断页面状态是否符合预期
  return screenshots;
}

module.exports = { validateFeature };
```

验证步骤配置示例：

```json
{
  "featureId": "auth/login",
  "steps": [
    { "name": "open_login", "url": "http://localhost:3000/login", "action": "none" },
    { "name": "fill_email",  "url": "http://localhost:3000/login", "action": "fill", "selector": "#email", "value": "test@example.com" },
    { "name": "after_submit", "url": "http://localhost:3000/login", "action": "click", "selector": "button[type=submit]" }
  ]
}
```

AI 拿到截图后，可以判断「登录成功后有没有跳转到 dashboard」「报错信息有没有正确显示」，而不是靠猜。这是 Harness 里验证层次最高的一环。

## 可理解环境（Legible Environment）

让 AI 能可靠地自主工作，有一个常被忽视的前提：**AI 必须能读懂它所在的环境**。工具再完善，AI 如果搞不清楚「这个项目是干什么的、哪些文件做什么用、哪些事情绝对不能做」，就会走冤枉路甚至搞破坏。

「可理解环境」（Legible Environment）的核心原则是：**仓库即真理（Repository as Truth）**。所有规范、约束、架构决策都必须以结构化文本的形式存在于仓库里，而不是藏在团队成员的脑子里或者某个 Confluence 页面里。

具体要做的事：

**1. AGENTS.md 索引**

在项目根目录放一个 `AGENTS.md`，这是专门给 AI 读的说明文档。它不是给人看的 README，内容更偏向约束和操作规范：

```markdown
# AGENTS.md — AI 智能体工作指引

## 项目结构
- src/api/      → REST API 路由，只处理 HTTP 层，业务逻辑放 services/
- src/services/ → 业务逻辑，不能直接访问数据库，通过 repositories/ 层
- src/models/   → 数据模型定义，不含业务逻辑

## 禁止事项
- 不得在 API 路由里直接写 SQL
- 不得修改 harness/ 目录下的任何文件
- 不得删除已通过的测试用例
- 不得使用 any 类型（TypeScript 项目）

## 代码风格
- 使用 async/await，禁止 .then().catch() 链式写法
- 错误统一通过 AppError 类抛出
- 所有公共函数必须有 JSDoc 注释

## 任务执行规范
- 每次只修改与当前任务直接相关的文件
- 完成后运行 npm test 确认没有破坏已有功能
- 把完成状态更新到 harness/features.json
```

**2. 结构化文档目录**

```
docs/
├── architecture.md   # 系统架构图和层级关系
├── api-spec.md       # 所有接口的入参出参规范
├── constraints.md    # 技术选型约束（为什么用 X 不用 Y）
└── error-codes.md    # 错误码对照表
```

这些文档的受众主要是 AI，所以格式比人类友好的叙述文字更重要。能用表格就用表格，能用代码示例就用示例，少写「请注意……」这种废话。

## 自定义 Linter + ArchUnit 架构测试

标准的 ESLint/Pylint 只能检查语法和风格，但 Harness 需要的是**业务层面的约束**：「API 层不能直接调数据库」「所有服务必须依赖注入」「不能引入未经审批的第三方库」。这些靠标准 linter 做不到。

**自定义 Linter 规则示例（ESLint 插件）：**

```javascript
// harness/linter-rules.js
module.exports = {
  rules: {
    // 禁止在 api/ 目录下直接 import db 相关模块
    'no-db-in-api': {
      create(context) {
        return {
          ImportDeclaration(node) {
            const filename = context.getFilename();
            const importPath = node.source.value;

            if (
              filename.includes('/api/') &&
              (importPath.includes('database') || importPath.includes('prisma'))
            ) {
              context.report({
                node,
                message: 'API 层不允许直接访问数据库，请通过 services/ 层调用'
              });
            }
          }
        };
      }
    },

    // 禁止 console.log 出现在非调试文件里
    'no-console-in-prod': {
      create(context) {
        return {
          CallExpression(node) {
            if (
              node.callee.type === 'MemberExpression' &&
              node.callee.object.name === 'console' &&
              !context.getFilename().includes('.test.')
            ) {
              context.report({ node, message: '生产代码不允许 console.log，请使用 logger' });
            }
          }
        };
      }
    }
  }
};
```

**ArchUnit 风格的架构测试（Jest 实现）：**

```javascript
// tests/arch.test.js
const glob = require('glob');
const fs = require('fs');

describe('架构约束测试', () => {
  test('API 路由不能直接引用数据库', () => {
    const apiFiles = glob.sync('src/api/**/*.js');
    apiFiles.forEach(file => {
      const content = fs.readFileSync(file, 'utf-8');
      expect(content).not.toMatch(/require.*database|import.*prisma/);
    });
  });

  test('Services 层必须有对应的测试文件', () => {
    const services = glob.sync('src/services/**/*.js')
      .filter(f => !f.includes('.test.'));
    services.forEach(service => {
      const testFile = service.replace('src/services', 'tests/services')
                               .replace('.js', '.test.js');
      expect(fs.existsSync(testFile)).toBe(true);
    });
  });

  test('不允许循环依赖', () => {
    // 可以用 madge 库检查
    const { execSync } = require('child_process');
    const result = execSync('npx madge --circular src/', { encoding: 'utf-8' });
    expect(result.trim()).toBe('');
  });
});
```

这类测试直接挂在 CI 流程里，AI 每次提交代码都必须跑一遍。架构腐化在第一步就被拦下来，而不是等到 code review 时才发现。

## 垃圾回收：对抗 AI 废料的具体实现

AI 生成的代码有一个特征：它倾向于复制当前文件里看到的模式，不管那个模式对不对、过没过时。几轮下来，代码库里会出现三种风格的日期处理、两套错误处理机制、一堆没人调用的 helper 函数。

这就是 AI 废料（AI slop）。Harness 需要一个定期跑的垃圾回收器来对抗这种熵增。

```javascript
// harness/garbage-collector.js
const { execSync } = require('child_process');
const fs = require('fs');
const glob = require('glob');

async function collectGarbage(aiClient) {
  const report = [];

  // 1. 找出未被引用的导出函数
  const unusedExports = execSync(
    'npx ts-unused-exports tsconfig.json',
    { encoding: 'utf-8' }
  ).trim();

  if (unusedExports) {
    report.push({ type: 'unused_exports', detail: unusedExports });
  }

  // 2. 找出重复的工具函数（简单版：相似函数名）
  const utilFiles = glob.sync('src/utils/**/*.js');
  const funcNames = [];
  utilFiles.forEach(file => {
    const content = fs.readFileSync(file, 'utf-8');
    const matches = content.matchAll(/function (\w+)/g);
    for (const m of matches) funcNames.push({ name: m[1], file });
  });

  // 找同名函数
  const nameCount = {};
  funcNames.forEach(({ name }) => {
    nameCount[name] = (nameCount[name] || 0) + 1;
  });
  const duplicates = funcNames.filter(({ name }) => nameCount[name] > 1);
  if (duplicates.length) {
    report.push({ type: 'duplicate_functions', detail: duplicates });
  }

  // 3. 把报告发给 AI 让它清理
  if (report.length > 0) {
    const prompt = `以下是代码库里发现的废料，请逐一清理：\n${JSON.stringify(report, null, 2)}`;
    const cleanupPlan = await aiClient.generate(prompt);
    fs.writeFileSync('harness/gc-report.json', JSON.stringify({
      date: new Date().toISOString(),
      items: report,
      plan: cleanupPlan
    }, null, 2));
  }

  return report;
}

module.exports = { collectGarbage };
```

垃圾回收器不需要每次 AI 运行后都触发，可以设成 cron job，每天晚上跑一次，把报告写到文件里，第二天开发开始前先处理废料再干正事。

## 自己写 vs 用现成框架

这个问题没有标准答案，取决于你的目标。

| 维度 | 自己写 | 用现成框架（LangGraph/AutoGPT/CrewAI） |
|------|--------|----------------------------------------|
| 上手速度 | 慢，需要从零搭建 | 快，pip install 就能跑 |
| 灵活度 | 完全可控，想怎么改就怎么改 | 受框架设计限制，深度定制麻烦 |
| 理解深度 | 强迫你搞懂每一个环节 | 容易停留在「用了但不懂」的状态 |
| 维护成本 | 自己承担，但代码是自己的 | 依赖框架升级节奏，有时 breaking change 让人头疼 |
| 适合场景 | 个人项目、高度定制需求、想学原理 | 团队快速落地、原型验证、标准化需求 |
| 生产可靠性 | 取决于你写得好不好 | 框架本身经过更多测试 |

**建议的路径**：先自己写一个简单版，把任务队列→检查→记忆这个循环跑通，哪怕只有 50 行代码。理解了原理之后，再根据需要选框架。跳过这一步直接用框架，很容易在出问题的时候不知道该怎么调。

现成框架的安装入口：

```bash
# Python 生态
pip install langgraph          # LangGraph，状态机风格，适合复杂工作流
pip install crewai             # CrewAI，多智能体协作

# Node.js 生态
npm install @openai/agents-sdk  # OpenAI 官方 Agents SDK
npm install langchain           # LangChain JS 版
```

值得注意的是，**Harness 是一个概念，不是一个产品**。你用 Shell 脚本实现它，和用 LangGraph 实现它，本质上做的是同一件事。框架帮你省掉了一些胶水代码，但核心逻辑——任务分解、检查、记忆、重试——还是你来设计。

---

## 小结

Harness 的构建方式从复杂到简单都行得通，关键是把这个循环打通：

```
任务队列给出下一步
    ↓
AI 执行并生成结果
    ↓
检查器验证（Linter + 测试 + 架构约束 + E2E）
    ↓
不通过 → 注入错误信息 → AI 重写
通过   → 写入记忆库 → 更新 features.json → 推进下一步
```

目录结构、脚本语言、框架选型都是形式，以上这个循环才是 Harness 的实质。搭好这个循环，AI 在长时任务里的可靠性会有明显提升——不是因为模型变聪明了，而是因为跑偏的代价变小了、纠错的速度变快了。
