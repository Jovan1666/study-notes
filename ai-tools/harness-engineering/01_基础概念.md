# Harness Engineering 基础概念

---

## 1. Harness 是什么？两个层面都要搞清楚

"Harness" 这个词在 AI 工程语境里有两个根源，一个来自软件测试领域，一个来自 AI 智能体工程领域。这两个概念不是同一个东西，但底层逻辑一脉相承——都是在给代码（或 AI）套一个可控、可管、有护栏的运行环境。

### 1.1 Test Harness（测试床）

这是软件测试领域早就有的概念。Test Harness 是一组桩程序（Stubs）和驱动程序（Drivers）的集合，专门用来给开发中的代码提供「钩子」（Hook），使其可以在自动化框架下独立运行和验证，不需要真实的生产环境。

说白了，就是给你的代码搭一个「模拟考场」：生产数据库没上线没关系，测试床帮你模拟一个；外部 API 不稳定没关系，测试床给你造一个假的桩。这样就能在隔离环境里跑自动化测试，发现问题后直接打回去改，不用等整套系统都就绪。

### 1.2 Agent Harness（智能体线束）

Agent Harness 是更新的概念，针对的是 AI 智能体（Agent）在长时间、多步骤任务中跑偏的问题。

它被定义为：**包裹在 AI 模型外围、专门用于管理和管控长时运行任务的软件基础设施。**

注意几个关键点：
- 它不是 AI 模型本身
- 它也不是一个开发框架（Framework）
- 它是模型「外围」的管控层，负责规范、引导、纠偏 AI 智能体的全生命周期运行

核心目标是解决「模型漂移」（Model Drift）问题——当 AI 需要连续执行数百次工具调用、持续运行数小时乃至数天的复杂工作流时，如何保证它不跑偏、不发疯、不产生大量垃圾代码。

### 1.3 两者对比

| 类型 | 定位 | 核心功能 |
|------|------|----------|
| Test Harness | 软件测试的模拟环境 | 桩（Stubs）+ 驱动（Drivers）的集合，为代码提供 Hook，让自动化测试在隔离环境中运行 |
| Agent Harness | AI 智能体的运行管控层 | 包裹在模型外围的基础设施，负责管理长时任务、强制质量标准、清理废料、对抗模型漂移 |

## 2. CPU / 内存 / OS 的类比

理解 Agent Harness 最直接的方式，就是用计算机架构来类比：

| 计算机组件 | 对应 AI 组件 | 作用 |
|-----------|-------------|------|
| CPU | AI 模型（如 Claude、GPT-4o） | 提供基础的计算和推理能力 |
| 内存（RAM） | 上下文窗口（Context Window） | 临时存储当前对话和任务的信息 |
| 操作系统（OS） | Agent Harness | 管理资源、提供标准驱动、保证环境稳定 |

CPU 再强，没有操作系统你也没法直接在上面跑复杂软件。裸 CPU 不知道怎么管内存、怎么调度进程、怎么处理 I/O 异常。操作系统就是那层把硬件能力变成可用服务的基础设施。

AI 模型也一样。模型本身提供推理能力，但它不知道：
- 当前任务进行到第几步了
- 上一步的输出有没有通过质量检查
- 代码库里积攒了多少技术债要清理
- 出错了应该怎么回滚重试

Harness 就是那层「操作系统」，把这些管理工作全接过来，让 AI 只需要专注于推理和生成，而不用担心环境问题。

## 3. Harness Engineering 是什么？

Harness Engineering（线束工程）是一套工程实践体系，核心任务是：**设计环境、脚手架和反馈循环，使 AI 智能体能够可靠地自主完成软件工作。**

这个定义有三个关键词：
- **环境**：AI 工作的场所，包括它能看到什么信息、能调用什么工具
- **脚手架**：支撑 AI 运行的结构，包括任务分解方式、进度记录机制
- **反馈循环**：AI 干完活之后有没有人/机制去检查，检查不通过怎么处理

Harness Engineering 包含三大支柱，缺一不可。

## 4. 三大支柱详解

### 4.1 上下文工程（Context Engineering）

上下文工程不是「给 AI 提供一个大的知识库」这么简单。它的核心是：**让 AI 在执行任务时能动态获取它真正需要的信息，而不是只靠静态文档。**

静态知识库的问题在于——AI 开发代码时，真正需要的往往不是「这个系统的设计文档」，而是：
- 当前这段代码跑起来报了什么错（日志）
- 页面上哪个按钮点了没反应（浏览器截图）
- 某个 API 调用的实时返回值是什么（终端输出）

所以上下文工程要做的是：
- 为 AI 提供可观测性工具的访问能力（如 LogQL 查日志、PromQL 查指标）
- 接入 Chrome DevTools 协议，让 AI 可以操控浏览器、截图、分析 DOM 树
- 设置 `progress.txt` 类的进度记录文件，让 AI 知道当前任务做到哪一步了
- 把任务分解成数百个微小特征，让每一步的上下文都保持干净、可追踪

用一句话概括：上下文工程是给 AI 配「实时仪表盘」，不是给它塞「百科全书」。

### 4.2 架构约束（Architectural Constraints）

架构约束的核心逻辑是：**不要靠人工代码审查来保证质量，要靠机器自动拦截。**

AI 生成代码的速度远超人类审查速度。你让 AI 跑一个中型项目，它半小时能产出几千行代码，这时候靠人去看每一行根本不现实。而且 AI 有一个习惯——它会「抄近路」，会把代码库里已有的旧模式复制放大，不管那个旧模式当初是不是一个临时方案。

架构约束的实现方式：

**1. 自定义代码检查器（Custom Linters）**

编写针对项目架构的自定义规则，比如：
- 「禁止在 Service 层直接调用 Repository 层以外的数据库操作」
- 「所有对外 API 必须经过统一的错误处理中间件」

这些规则通过 Git pre-commit Hook 强制执行。AI 提交代码前，Linter 自动扫描，不合格直接拒绝，并把错误报告返回给 AI，要求重新生成。

**2. 结构化测试（Structural Tests）**

使用 ArchUnit 等工具，不是测「功能对不对」，而是测「架构对不对」。例如：
- Controller 层是否只依赖 Service 层
- 某个核心模块是否没有引入不该有的外部依赖

**3. 机械化拦截**

这是架构约束的精髓：AI 想走捷径写出混乱的依赖关系，Harness 直接拦截操作，把错误信息甩回给 AI，要求重新生成合规代码。整个过程不需要人介入，完全自动化。

一个工程实践案例值得一提：Vercel 的团队发现，移除 80% 的手工专用工具，改用通用工具 + 架构约束后，AI 的任务成功率从 80% 提升到了 100%。过度定制化的工具反而让 AI 更容易出错。

### 4.3 垃圾回收（Garbage Collection）

垃圾回收针对的是「AI 废料」（AI Slop）的积累问题。

AI 在长期开发过程中，会不断复制并放大代码库里的旧模式，导致：
- 重复的代码模式到处都是
- 架构设计越来越不一致
- 冗余代码越来越多，但没有人清理

这些废料积累到一定程度，会形成「代码熵增」——系统越来越难维护，AI 后续生成的代码质量越来越差，因为它在用一个已经混乱的代码库作为参考。

垃圾回收的实现方式：

- **后台监控智能体**：专门部署一个「巡逻」用的 AI 智能体，定期扫描代码库，发现偏离架构规范的部分
- **自动重构 PR**：发现问题后，自动开启一个 Refactoring PR（重构合并请求），描述清楚问题和改法，等待人工确认合并
- **技术债实时偿还**：把技术债看作「高息贷款」，不拖到项目末期一次性还，而是在每次增量开发中持续小额偿还

## 5. 模型漂移（Model Drift）

模型漂移是 Harness Engineering 要解决的核心问题之一，理解它对理解 Harness 的必要性至关重要。

### 什么是模型漂移？

模型漂移指的是：AI 智能体在执行长时间、多步骤任务时，随着对话轮次增加、上下文积累，其行为和输出质量逐渐偏离初始目标的现象。

具体表现：
- 前 200 步逻辑清晰、输出高质量；第 300 步开始出现小错误；第 400 步开始前后矛盾；第 500 步彻底「发疯」，开始重写之前已经正确的代码
- AI 忘记了任务的整体目标，开始在局部细节上打转
- 对同一个问题，不同步骤给出互相矛盾的解答
- 开始产生「幻觉」，引用不存在的函数或文件

### 为什么会漂移？

根本原因在于上下文窗口（Context Window）是有限的。当任务足够长，早期的关键约束、架构决策和任务目标会被「挤出」上下文窗口，AI 实际上是在「失忆」状态下继续工作的。

另一个因素是：顶级模型在静态评测榜单上的差距已经不足 1%，但在处理数百步逻辑的长时任务时，哪怕是最强模型也会在中间步骤出现质量断崖。这不是某个模型的问题，是当前所有模型的共同限制。

### Harness 如何应对漂移？

```
无 Harness：第 200 步开始跑偏 → 第 400 步彻底发疯
有 Harness：第 200 步被检查打回 → 强制重试 → 第 500 步才开始疲软
```

Harness 不能彻底消除漂移，但能通过**强制分段 + 及时纠错**的机制，把漂移发生的时间点显著推后，并在漂移发生时自动触发纠正流程。

## 6. Harness 与「写文档/写 Rules」的核心区别

很多人（包括我自己）走过这条路：先写详细的开发规范文档，再写 .cursorrules 或 CLAUDE.md，给 AI 注入约束，然后发现问题依然存在。

这是因为文档和 Rules 是**静态的**，而 Harness 是**动态执行的**。

| 维度 | 写文档 / 写 Rules | Harness |
|------|------------------|---------|
| 性质 | 静态文本，被动被读取 | 动态执行，主动触发 |
| 执行时机 | AI 读取后不一定遵守 | 在关键节点自动拦截和验证 |
| 违规处理 | AI 可能「理解」了但不执行 | 违规直接被技术手段阻断，无法跳过 |
| 随时间的有效性 | 随上下文增加逐渐被「遗忘」 | 每次提交/每个步骤都重新触发检查 |
| 维护成本 | 文档越写越多，越难维护 | 规则代码化，和代码库一起版本管理 |
| 对 AI 漂移的抵抗力 | 几乎没有，漂移后直接失效 | 有，每个检查点都是一次「重新校准」 |

一句话说清楚区别：**文档是告诉 AI「应该怎么做」，Harness 是让 AI「不能不这么做」。**

## 7. Hook（钩子）完整解释

Hook 这个词在 Harness 语境里有两种具体指代，不能混用。

### 7.1 测试床 Hook（Test Harness Hook）

在传统软件测试（Test Harness）的语境里，Hook 是指为开发代码预留的**标准化接入点**。

它的作用是：允许自动化测试框架「钩住」并直接调用你代码里的特定函数或组件，不需要启动整个生产系统。测试系统通过这个 Hook 输入预设参数，运行目标代码，捕获输出结果，和预期值对比。

形象地说：就像在机器人的手臂上预留了一个**标准插孔**。不用把整个机器人拆开，把测试仪器插进这个插孔，就能单独测试这只手臂的力度和反应。

### 7.2 生命周期 Hook（Lifecycle Hook）

在 Agent Harness Engineering 的语境里，Hook 更多指的是**生命周期钩子**，也叫流程触发器。

最典型的例子是 Git 的 pre-commit Hook：

```bash
# 你运行: git commit -m "修复登录 bug"
#
# pre-commit hook 在提交之前自动触发：
#   → 运行自定义 Linter，检查架构合规性
#   → 运行结构化测试
#   → 检查不通过 → 提交被自动取消
#   → 错误报告返回给 AI → AI 重新生成代码
```

React 里的 `useEffect` 也是一种 Hook——在组件「挂载到页面」这个生命周期节点自动触发执行。

生命周期 Hook 的本质：**在某个预定的时机，自动触发某个检查或操作，不需要人手动介入。**

形象地说：就像工厂传送带上安装的红外线感应器。AI 工人把零件组装好传过来，感应器（Hook）自动触发检查。零件尺寸不对，感应器立刻「钩住」它不让通过，并要求 AI 工人重做——全自动，不需要质检员盯着。

### 7.3 两种 Hook 的对比

| 类型 | 场景 | 触发时机 | 核心作用 |
|------|------|----------|----------|
| 测试床 Hook | 软件测试 | 测试框架主动调用 | 提供标准化接入点，隔离测试目标代码 |
| 生命周期 Hook | 智能体管控 | 特定流程节点自动触发 | 强制执行质量标准，拦截不合规输出 |

## 8. AI 废料（AI Slop）

### 什么是 AI 废料？

AI Slop（AI 废料）是指 AI 智能体在长期开发过程中，通过复制、放大代码库里已有的旧模式或次优模式，逐渐堆积出来的低质量代码。

「Slop」这个词在英语里本来是「泔水、猪食」的意思，用来形容那些看起来能跑但实际上一团糟的代码非常贴切。

具体表现：
- **重复的代码模式**：同样的逻辑在不同文件里被写了三四遍，没有抽象复用
- **不一致的架构设计**：A 模块用了一套错误处理风格，B 模块用了另一套，C 模块完全不处理错误
- **冗余的依赖引入**：AI 不确定有没有某个工具函数，直接自己写一个，结果代码库里现在有三个功能一样的工具函数
- **失控的注释和文档字符串**：AI 生成了大量格式化注释，但内容和实际代码不符
- **已废弃模式的复活**：早期代码库里有个临时方案，早该删了但没删，AI 把它当成「标准做法」大量复制

### 为什么危害大？

AI 废料的危害是指数级扩大的，不是线性的：

1. **正反馈循环**：AI 下次生成代码时，会参考当前代码库。废料越多，AI 越倾向于生成更多废料，形成恶性循环
2. **漂移加速**：混乱的代码库让模型漂移来得更快，因为 AI 的「参考系」本身就是错的
3. **维护成本暴增**：人工审查越来越难，因为代码量大、模式不一致
4. **技术债利滚利**：每一行废料代码都在增加后续修改的难度，就像高息贷款，拖得越久代价越高

### Harness 如何清理 AI 废料？

通过垃圾回收机制（见第 4.3 节），部署专门的后台监控智能体定期扫描，发现偏离架构的部分自动开启重构 PR。关键是把清理工作**自动化、持续化**，而不是等到项目末期再来一次大扫除。

## 9. 成熟度模型 Level 1–5

如何评估一个团队在 Harness Engineering 上的成熟度？这里有一个 5 级模型，可以用来自测。

### Level 1：文档化（Documentation）

**特征**：有基础的开发规范文档，AI 使用有一定约束，但全靠人工执行。

- 有 .cursorrules、CLAUDE.md 或类似的规范文件
- 有代码风格指南，但靠人工审查来保证执行
- AI 生成的代码需要人逐行检查
- 没有自动化的质量门禁

**瓶颈**：规范文档随时间膨胀，AI 越来越难「读完」，执行率越来越低。

### Level 2：自动化门禁（Automated Gates）

**特征**：代码提交前有自动化检查，基础质量有机器保证。

- 配置了 Git pre-commit Hook，提交前自动运行 Linter 和单元测试
- 引入 CI/CD 流水线，合并 PR 前必须通过自动化测试
- 有基础的架构规则检查
- AI 生成的代码如果不通过检查，无法提交

**瓶颈**：AI 可以看到日志吗？AI 能自己去查哪里出错了吗？还不行。

### Level 3：环境对智能体透明（Legible Environment）

**特征**：AI 可以自主查询运行环境的状态，不需要人类「翻译」错误信息给它。

- 为每个 AI 智能体分配隔离的应用实例
- AI 可以访问日志查询工具（如 LogQL）和指标查询工具（如 PromQL）
- 接入 Chrome DevTools 协议，AI 可以自己打开浏览器、截图、分析 DOM
- 设有 progress.txt 类的进度文件，AI 知道任务执行到哪一步了

**瓶颈**：AI 发现问题了，但还是需要人来决定「要不要修」「怎么修」。

### Level 4：智能体驱动的自主循环（Agent-Driven Loops）

**特征**：AI 不只能发现问题，还能自主触发修复循环。

- AI 发现测试失败 → 自动分析错误日志 → 自主生成修复代码 → 自动重新提交验证
- 后台垃圾回收智能体开始运行，定期扫描代码库并开启重构 PR
- 人的角色开始从「执行者」转变为「审核者」
- 大多数常规 Bug 不再需要人介入

**瓶颈**：复杂的架构决策、跨模块的逻辑串联，AI 还撑不住。

### Level 5：自我修复（Self-Healing）

**特征**：全自主的反馈闭环，系统具备自我修复能力，平均修复时间缩短至分钟级。

- Harness 捕捉到的失败点自动成为优化下一代模型的训练素材
- 系统可以识别自己当前处于哪个「漂移阶段」，主动触发重置或重构
- 人只需要设定目标和审核最终结果，中间过程完全自动化
- 技术债实时偿还，代码库长期保持健康状态

**现状**：Level 5 目前还是前沿方向，大多数工程团队处于 Level 2 到 Level 3 之间。

## 10. 关键认知：Harness 不是银弹

Harness Engineering 能做什么，不能做什么，要说清楚。

**能做的**：
- 延缓模型漂移，把「AI 发疯」的时间点显著推后
- 通过机械化约束保证基础代码质量
- 减少人工介入的频率，让工程师从「AI 保姆」变成「目标设定者」
- 对抗代码熵增，保持代码库长期健康

**不能做的**：
- 彻底解决模型的长程推理能力限制——这是模型本身的问题，不是工程问题
- 替代工程师做跨模块的全局架构决策
- 让 AI 独立完成中大型项目——至少目前的模型做不到

现实的分工：**AI 负责「小块确定性强的工作」（写函数、调 Bug、写组件），工程师负责「大块需要全局观的串联工作」（定接口、做架构决策、串联模块）。** 这是目前最有效的人机协作模式。

Harness 是让 AI 多用一段时间、少出乱子的工具，不是让 AI 无限工作不出错的魔法。
