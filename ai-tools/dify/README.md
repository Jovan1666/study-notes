# Dify 低代码 AI 应用开发

> 来源：[langgenius/dify](https://github.com/langgenius/dify) & [svcvit/Awesome-Dify-Workflow](https://github.com/svcvit/Awesome-Dify-Workflow)

Dify 是一款开源的大语言模型应用开发平台，融合了后端即服务（BaaS）和 LLMOps 的理念。平台提供从原型设计到生产部署的完整闭环，支持通过 YAML 文件或可视化界面定义 AI 应用的提示词、上下文及插件。

## Dify 是什么

### 核心技术能力

- **RAG 引擎**：内置检索增强生成管道，支持从文档中提取知识
- **Agent 框架**：基于大模型的智能体，可调用外部工具
- **工作流编排**：可视化编排复杂的 AI 处理流程
- **模型中立**：支持接入 GPT、Claude、Llama、通义千问等数百种模型

### 设计哲学

Dify 的名字来自 **Define**（定义）和 **Modify**（改进），目标是为开发者提供 **Do It For You** 的体验——让 AI 开发变得简单、直观、高效。

### 通俗理解

把 Dify 想象成一个「AI 应用的拼装工具箱」。

| 方案 | 比喻 |
|------|------|
| 传统方式（LangChain 等） | 自己买锤子钉子，从零开始敲打 |
| Dify | 预制房支架已搭好，只需布置房间、搬入家具 |

### 部署方式

```bash
cd dify/docker
cp .env.example .env
docker compose up -d
```

访问 `http://localhost/install` 完成初始化。

## Chatflow 与 Workflow

Dify 中有两类工作流应用，选对类型能事半功倍。

### 一图对比

| 特性 | 对话流（Chatflow） | 工作流（Workflow） |
|------|-------------------|-------------------|
| 核心目标 | 交互式对话体验 | 任务自动化与批处理 |
| 对话记忆 | 支持多轮上下文 | 无，单次执行 |
| 结束方式 | 回复节点持续交互 | 结束节点终止流程 |
| 典型场景 | 智能客服、AI 伴侣 | 报表生成、内容创作 |

### 对话流（Chatflow）

**核心特征**：多轮对话（Memory）、流式输出、可配置开场白和快捷问题

**适用场景**：智能客服、教育辅导、需要不断追问的语义搜索

### 工作流（Workflow）

**核心特征**：线性执行、无状态、高效吞吐

**适用场景**：内容创作流水线、自动化批处理、定时任务

**条件分支示例**：Workflow 支持通过「条件节点」实现分支逻辑

![Dify 条件分支示例](./assets/dify-workflow-condition-demo.png)

### 选择决策树

```
是否需要与用户持续对话？
├── 是 → 是否需要记忆上下文？
│   ├── 是 → Chatflow
│   └── 否（单轮问答）→ 两者皆可
└── 否 → 流程是否固定？
    ├── 是 → Workflow
    └── 否（需要对话调整）→ Chatflow
```

**简化判断**：
- 想做「聊天机器人」→ Chatflow
- 想做「点一下按钮，自动完成一系列操作」→ Workflow

## 工作流设计思路

### 核心观点

用 Dify 构建应用，最关键的不是写代码，而是**把业务流程想清楚**。

### 设计流程 vs 实现流程

Dify 本质上是一个「编排工具」，不是「编程工具」。

**传统思路**：拿到需求 → 写代码实现 → 调试修 bug

**推荐思路**：先把业务流程在脑子里跑通 → 用 LLM 辅助梳理盲点 → 照着流程图去 Dify 里拼节点

### 用强模型辅助设计

新手常遇到的问题：自己想的流程有疏漏，等搭完了才发现逻辑不对。

**建议**：在设计阶段就用强模型（Claude Opus/GPT-4）帮你审视流程。

可以这样描述你的需求：
> 我想做一个「学生成绩查询」的工作流，老师输入学号，返回该学生的各科成绩。请帮我检查这个流程设计有没有遗漏。

让模型指出：
- 异常情况如何处理（学号不存在怎么办）
- 需不需要权限验证
- 返回格式是否清晰

### 变量设计的处理

**变量的本质**：每个节点只需要知道「我从谁那里拿数据」、「我把数据交给谁」，范围其实很有限。

**实用技巧**：
1. 先画节点再配变量，不要一开始就想全局变量
2. 善用「变量聚合器」节点合并多路分支的输出
3. 变量命名要「所见即所得」

### 场景专一原则

同一个工作流，只做同一类事。

| 适合 | 不适合 |
|------|--------|
| 查询成绩 + 查询学籍 + 查询任教班级 | 查询成绩 + 查询周边房价 |

### 技术选型建议

| 需求 | 推荐方案 |
|------|---------|
| 查制度文档、合同、规章 | RAG 知识库检索 |
| 查成绩、学籍等结构化数据 | HTTP 请求节点 + 外部 API，或 Code 节点写 SQL |

### 代码节点的定位

代码节点在 Dify 里是「胶水」，不是主力。**让 LLM 写代码节点**完全可以——用 GPT-4 或 Claude 描述清楚逻辑，让模型生成代码，比手写效率高得多。

## 可用模板

来源：[Awesome-Dify-Workflow](https://github.com/svcvit/Awesome-Dify-Workflow)，使用 Dify 0.13.0 及以上版本导入。

### Agent 与工具调用

| 文件名 | 说明 |
|--------|------|
| `AgentFlow.yml` | Agent 基础流程示例 |
| `Agent工具调用.yml` | 工具调用核心逻辑 |
| `MCP.yml` / `MCP-amap.yml` | MCP 工具集成 |

### 搜索与研究

| 文件名 | 说明 |
|--------|------|
| `Deep Researcher On Dify.yml` | 深度研究助手 |
| `搜索大师.yml` | 网页搜索增强 |

### 翻译与语言处理

| 文件名 | 说明 |
|--------|------|
| `translation_workflow.yml` | 通用翻译 |
| `中译英.yml` / `英译中.yml` | 双向翻译 |
| `全书翻译.yml` | 整本文档翻译 |

### 内容创作

| 文件名 | 说明 |
|--------|------|
| `标题党创作.yml` | 标题生成 |
| `文章仿写-单图_多图自动搭配.yml` | 仿写 + 配图 |

### 互动游戏

| 文件名 | 说明 |
|--------|------|
| `完蛋！我被LLM包围了！.yml` | 热门互动游戏 |
| `完蛋！我被LLM包围了！（战绩排行版）.yml` | 带排行榜版 |

### 注意事项

1. 部分模板使用 Agent 节点 1.0 版本功能，请使用 Dify 0.13.0 及以上
2. 导入后需配置所使用的模型 API
