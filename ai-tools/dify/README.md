# Dify 低代码 AI 应用开发

> 来源：[langgenius/dify](https://github.com/langgenius/dify) & [svcvit/Awesome-Dify-Workflow](https://github.com/svcvit/Awesome-Dify-Workflow)

Dify 是一款开源的大语言模型应用开发平台，融合了后端即服务（BaaS）和 LLMOps 的理念。平台提供从原型设计到生产部署的完整闭环，支持通过 YAML 文件或可视化界面定义 AI 应用的提示词、上下文及插件。

## Dify 是什么

### 核心技术能力

- **RAG 引擎**：内置检索增强生成管道，支持从文档中提取知识
- **Agent 框架**：基于大模型的智能体，可调用外部工具
- **工作流编排**：可视化编排复杂的 AI 处理流程
- **模型中立**：支持接入 GPT、Claude、Llama、通义千问等数百种模型

### 设计哲学

Dify 的名字来自 **Define**（定义）和 **Modify**（改进），目标是为开发者提供 **Do It For You** 的体验——让 AI 开发变得简单、直观、高效。

### 通俗理解

把 Dify 想象成一个「AI 应用的拼装工具箱」。

如果把开发 AI 应用比作盖房子：

| 方案 | 比喻 |
|------|------|
| 传统方式（LangChain 等） | 自己买锤子钉子，从零开始敲打 |
| Dify | 预制房支架已搭好，只需布置房间、搬入家具 |

你需要做的：
1. 选择一个基础模型（GPT、Claude、Llama 等）
2. 通过拖拽配置工作流程
3. 发布为网站或 API

### 能解决什么问题

**场景一：企业知识库问答**

将产品手册、规章制度等文档喂给 Dify，搭建一个精准回答内部问题的助手。AI 基于真实文档作答，不会凭空编造。

**场景二：智能体（Agent）**

让 AI 不仅会聊天，还能自主执行多步骤任务。例如让它写财务报告时，自动搜索数据、计算指标、生成图表。

**场景三：自动化工作流**

设置一个定时执行的流程：爬取新闻 → 翻译 → 总结 → 发送到邮箱，全程无需人工干预。

### 适合哪些人

| 人群 | 为什么适合 |
|------|-----------|
| 创业者 | 几天内做出 MVP，验证市场需求 |
| 开发者 | 通过 API 将 AI 能力嵌入现有系统 |
| 企业 | 私有化部署，保证数据安全 |
| 非技术人员 | 拖拽配置，无需写代码也能做 AI 应用 |

### 部署方式

```bash
cd dify/docker
cp .env.example .env
docker compose up -d
```

访问 `http://localhost/install` 完成初始化。

源码部署适合需要深度定制或二次开发的场景，详见 [官方部署文档](https://docs.dify.ai/getting-started/install-self-hosted)。

## Chatflow 与 Workflow

Dify 中有两类工作流应用，选对类型能事半功倍。

### 一图对比

| 特性 | 对话流（Chatflow） | 工作流（Workflow） |
|------|-------------------|-------------------|
| 核心目标 | 交互式对话体验 | 任务自动化与批处理 |
| 对话记忆 | 支持多轮上下文 | 无，单次执行 |
| 结束方式 | 回复节点持续交互 | 结束节点终止流程 |
| 典型场景 | 智能客服、AI 伴侣 | 报表生成、内容创作 |

### 对话流（Chatflow）

#### 核心特征

- **多轮对话**：内置 Memory 节点，AI 能记住之前的聊天内容
- **流式输出**：响应逐字显示，体验接近 ChatGPT
- **交互元素**：可配置开场白、快捷问题建议、对话框样式

#### 适用场景

- **智能客服**：用户不断追问，AI 根据上下文修正回答
- **教育辅导**：根据用户上一题的回答，生成针对性反馈
- **语义搜索**：用户通过多轮对话逐步缩小搜索范围

#### 不适用场景

流程相对固定、不需要用户持续输入的情况。例如每天自动生成报表，这种更适合 Workflow。

### 工作流（Workflow）

#### 核心特征

- **线性执行**：从开始到结束，流程确定
- **无状态**：每次运行只处理当前输入，不保留历史
- **高效吞吐**：适合批量处理、API 调用、复杂计算

#### 适用场景

- **内容创作流水线**：例如小红书运营工作流，输入主题 → 生成标题 → 写正文 → 配图全自动
- **自动化批处理**：文档翻译、数据清洗、格式转换
- **定时任务**：每天自动爬取新闻并汇总发送

#### 不适用场景

需要 AI 根据用户反馈调整输出的场景。例如 AI 教练功能，用户回答问题后需要 AI 继续追问深入了解——这种情况 Chatflow 更合适。

#### 条件分支示例

Workflow 支持通过「条件节点」实现分支逻辑，根据输入内容走向不同处理路径：

![Dify 条件分支示例](./assets/dify-workflow-condition-demo.png)

如图所示，系统根据「是否年满 18 岁」进行分支，对应不同的处理流程。

### 选择决策树

```
是否需要与用户持续对话？
├── 是 → 是否需要记忆上下文？
│   ├── 是 → Chatflow
│   └── 否（单轮问答）→ 两者皆可
└── 否 → 流程是否固定？
    ├── 是 → Workflow
    └── 否（需要对话调整）→ Chatflow
```

**简化判断**：
- 想做「聊天机器人」→ Chatflow
- 想做「点一下按钮，自动完成一系列操作」→ Workflow

### 实际案例

#### Chatflow 案例：企业内部知识库助手

```
用户：你们的年假是怎么计算的？
AI：普通员工满一年有5天年假。请问您入职多久了？
用户：2年半
AI：根据您的入职时间，您现在有12天年假，上半年还能用6天...
```

AI 根据用户追问动态调整回答内容，需要 Memory 维护上下文。

#### Workflow 案例：文章一键发布

```
输入主题
  ↓
LLM 生成标题 + 摘要
  ↓
LLM 生成正文
  ↓
生成配图描述 → 调用绘图 API
  ↓
输出完整文章包
```

每个步骤输入输出固定，适合自动化执行。

## 工作流设计思路

### 核心观点

用 Dify 构建应用，最关键的不是写代码，而是**把业务流程想清楚**。

### 设计流程 vs 实现流程

Dify 本质上是一个「编排工具」，不是「编程工具」。

**传统思路**：拿到需求 → 写代码实现 → 调试修 bug

**推荐思路**：先把业务流程在脑子里跑通 → 用 LLM 辅助梳理盲点 → 照着流程图去 Dify 里拼节点

流程逻辑想清楚后，具体每个节点怎么配置、代码节点写什么——这些反而是小问题。

### 用强模型辅助设计

新手常遇到的问题：自己想的流程有疏漏，等搭完了才发现逻辑不对。

**建议**：在设计阶段就用强模型（Claude Opus/GPT-4）帮你审视流程。

可以这样描述你的需求：
> 我想做一个「学生成绩查询」的工作流，老师输入学号，返回该学生的各科成绩。请帮我检查这个流程设计有没有遗漏。

让模型指出：
- 异常情况如何处理（学号不存在怎么办）
- 需不需要权限验证
- 返回格式是否清晰

### 变量设计的处理

变量多是新手最容易「晕掉」的地方。

#### 变量的本质

每个节点只需要知道「我从谁那里拿数据」、「我把数据交给谁」，范围其实很有限。

#### 实用技巧

**技巧一：先画节点再配变量**

不要一开始就想全局变量。先把流程图画出来，标出每个节点需要什么输入输出，再去 Dify 里配置。

**技巧二：善用变量聚合器**

当多个分支的输出需要汇合时，用「变量聚合器（Variable Aggregator）」节点统一处理，避免在下游节点里逐个引用。

**技巧三：变量命名要「所见即所得」**

用 `student_id` 就别又改成 `stu_no`，在配置复杂流程时命名混乱会极大增加调试难度。

### 场景专一原则

同一个工作流，只做同一类事。

#### 好的例子

「教师成绩查询系统」：
- 查询学生成绩
- 查询学籍信息
- 查询任教班级

这三个场景业务背景相似，都是「教育管理信息查询」，可以放在同一个工作流里。

#### 不好的例子

「学校信息查询系统」里突然加入「查询周边房价」——业务背景完全不同，会导致提示词难以设计，输出质量也无法保障。

### 技术选型建议

#### RAG vs 数据库查询

| 需求 | 推荐方案 |
|------|---------|
| 查制度文档、合同、规章 | RAG 知识库检索 |
| 查成绩、学籍等结构化数据 | HTTP 请求节点 + 外部 API，或 Code 节点写 SQL |

**重要区别**：RAG 适合非结构化文档，数据库查询适合结构化数据。成绩查询背后通常是数据库，用 RAG 反而精度不够、响应不稳定。

#### 多场景共存

如果确实需要在一个应用里服务多个场景（比如既查成绩又查校规），使用「问题分类器」节点先识别意图，再分发到对应分支处理。

### 代码节点的定位

代码节点在 Dify 里是「胶水」，不是主力。

**代码节点擅长的事**：
- 数据格式转换（把 API 返回的 JSON 拆解提取）
- 复杂条件判断
- 简单计算

**让 LLM 写代码节点**：完全可以。用 GPT-4 或 Claude 把你想要的逻辑描述清楚，让模型生成代码，比手写效率高得多。

### 常见误区

| 误区 | 正确做法 |
|------|---------|
| 追求节点多、功能全 | 先做一个最小可用版本，再迭代 |
| 变量越多越好 | 变量越少越容易维护，每个变量必须有明确用途 |
| 所有场景塞进一个工作流 | 专一场景、专业处理 |
| 自己想不完善就开始搭 | 先用强模型审视流程逻辑 |

### 总结

工作流设计的核心：**业务逻辑先行，技术实现跟上**。

用「强力 AI 辅助设计流程 → LLM 生成代码节点 → 专一场景落地」这个模式，能显著提升搭出高质量应用的概率。
