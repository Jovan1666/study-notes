# 一、RAG 是什么？为什么要用它？

> 这不是教你写代码的教程，而是帮你搞清楚「为什么这样设计」的文档。

---

## 最直观的比喻：闭卷考试 vs 开卷考试

普通大语言模型（LLM）就像一个参加**闭卷考试**的学生。

它在训练阶段背了大量知识，考试时只能靠记忆作答。问题是：

- 训练数据有截止日期，最新的事它不知道
- 记忆可能出错，产生幻觉（Hallucination）
- 无法引用具体来源，说了也没法验证

RAG（Retrieval-Augmented Generation，检索增强生成）让模型变成**开卷考试**的学生。

考试前先去图书馆查资料，找到相关内容带进考场，再结合资料作答。

这样：
- 答案基于真实文档，不是凭空捏造
- 可以处理训练数据截止日期之后的信息
- 能够引用来源，答案可溯源

---

## RAG 解决的核心问题

| 问题 | 没有 RAG | 有 RAG |
|------|---------|-------|
| 知识时效性 | 只知道训练截止日前的事 | 可实时查询最新文档 |
| 幻觉问题 | 不确定时会编造 | 基于检索到的真实内容生成 |
| 私有知识 | 无法访问企业内部文档 | 可接入私有知识库 |
| 可溯源性 | 无法追溯答案来源 | 可标注引用来源 |

---

## 什么时候用 RAG

适合用 RAG 的场景：

- **企业知识库问答**：产品手册、规章制度、内部文档
- **客服系统**：基于真实产品信息回答用户问题
- **技术文档助手**：帮开发者快速找到 API 用法
- **法律/医疗咨询**：需要引用具体条文或资料的场景

不适合用 RAG 的场景：

- 纯粹的创意写作（不需要检索事实）
- 模型本身已经足够胜任的通用问答
- 文档量极少、直接塞进上下文更简单的情况
