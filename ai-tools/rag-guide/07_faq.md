# 七、常见问题

---

## Q：分片大小怎么定？

没有通用答案，取决于文档类型和使用场景。

**起点建议：**
- 中文文档：300-500 字/块，重叠 50 字
- 英文文档：512 token/块，重叠 50 token
- 技术文档（代码、表格多）：按语义边界切，不硬切

**调试思路：**
1. 先用默认值跑起来
2. 找几个典型问题，看召回结果质量
3. 如果召回内容缺乏上下文 → 增大分片
4. 如果召回内容噪音多 → 减小分片
5. 反复迭代

---

## Q：什么情况下需要重排序？

以下情况值得加重排序：

- 知识库文档量超过几千块
- 用户问题比较模糊或多义
- 向量检索召回结果质量不稳定
- 对答案精确度要求高（法律、医疗等场景）

小项目、文档量少的情况下，向量检索直接 top-3 已经够用，不需要重排序增加复杂度。

## Q：Embedding 模型怎么选？

**中文场景：**
- BAAI/bge-m3：综合效果最好，支持中英双语，免费开源
- text-embedding-3-small：OpenAI 出品，价格低，效果不错

**英文场景：**
- text-embedding-3-small / large：主流选择
- nomic-embed-text：开源，本地运行，适合隐私要求高的场景

**原则：** Embedding 模型和重排序模型最好来自同一系列（如都用 bge 系列），向量空间一致，效果更稳定。

---

## Q：怎么评估 RAG 效果？

**定性评估（快速验证）：**
- 准备 10-20 个典型问题
- 手动看召回内容是否相关
- 手动看答案是否正确、是否有编造

**定量评估（系统性）：**
- 构建评估数据集（问题 + 标准答案 + 参考文档）
- 用 RAGAS 框架自动计算召回率、忠实度、答案相关性等指标
- 对比不同配置下的指标变化

---

## Q：RAG 和 Fine-tuning 怎么选？

| 场景 | 推荐方案 |
|------|--------|
| 知识会频繁更新 | RAG |
| 私有文档、内部知识库 | RAG |
| 需要改变模型的「风格」或「行为」 | Fine-tuning |
| 知识固定、要求极低延迟 | Fine-tuning |
| 两者皆有需求 | RAG + Fine-tuning 结合 |

大多数企业知识库场景，RAG 是更合适的起点。Fine-tuning 成本高，且每次知识更新都要重新训练。
