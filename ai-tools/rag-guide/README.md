# 从零搞懂 RAG：一份写给自己的学习笔记

> 这不是教你写代码的教程，而是帮你搞清楚「为什么这样设计」的文档。代码随时可以问 AI，但设计思路才是真正需要自己消化的东西。

本文是我在学习 RAG（Retrieval-Augmented Generation）过程中整理的笔记，主要用 [NotebookLM](https://notebooklm.google.com) 做资料梳理，配合自己写的 Python 示例代码（见 [rag_system.ipynb](./rag_system.ipynb)）来加深理解。写这篇文章，是想把当时自己真正卡住过的地方说清楚，而不是复制一遍官方文档。

后续我也会尝试用 **Dify** 来实现真正可用的 RAG 工作流，那是另一篇的事了。

---

## 一、RAG 是什么？为什么要用它？

### 最直观的比喻

把 LLM 想象成一个准备闭卷考试的学生——它只能靠自己脑子里装的东西作答。问题很明显：

- 知识有截止日期，训练之后发生的事它不知道
- 不了解你的私有资料（公司文档、个人笔记）
- 不确定的地方容易「编」答案，也就是所谓的幻觉（Hallucination）

RAG 的思路是把闭卷改成开卷：**先去翻资料，再回答问题**。

```
RAG = Retrieval（检索）+ Augmented（增强）+ Generation（生成）
```

翻译成大白话：给 AI 挂一个外部知识库，它回答之前先去查，查到相关内容再作答。答案有出处，也不容易瞎编。

---

## 二、整体流程

![RAG 系统架构图](./assets/rag-architecture.png)

RAG 分两个阶段，逻辑很清晰：

```
第一阶段：准备阶段（建好书架，方便查阅）
  长文档 ──→ 分片 ──→ 转向量 ──→ 存入向量数据库

第二阶段：回答阶段（查书、筛选、作答）
  用户问题 ──→ 召回 ──→ 重排 ──→ 喂给 LLM ──→ 最终答案
                (粗选)   (精选)
```

两个阶段各自独立。准备阶段只做一次；回答阶段每次提问都会完整走一遍。

---

## 三、第一阶段：把文档整理好

### 3.1 分片（Chunking）

**为什么要分片？**

不能把整本书塞给 AI——上下文窗口有限，而且大量无关内容混在一起，AI 反而更容易答偏。所以要把文档切成小块，检索时只取相关的几块。

**分片的核心权衡：**

| 切太小 | 切太大 |
|--------|--------|
| 语义不完整，一句话被切成两半 | 包含太多无关内容，干扰 AI 判断 |
| 检索出来的片段缺少上下文 | 精度下降，噪音变多 |

经验值：中文约 100~200 字一块，英文约 300~500 个字符，实际要根据文档类型调整。

**两种常用策略：**

1. **固定长度分片** — 简单粗暴，每 N 个字符切一刀。适合无明显结构的文本，比如聊天记录。
2. **语义单元分片** — 按段落、章节切。适合有结构的文档，比如教程、报告。

还有一个细节：**加重叠（overlap）**。相邻两块之间保留一些共同内容，防止一个概念恰好被切在边界上、两边各讲了一半。

### 3.2 向量化（Embedding）

文本没法直接比较相似度，需要先转成一串数字（向量）。这个过程叫 Embedding。

核心是：**语义相近的文本，转出来的向量在数学上也更接近。**

```
「Python 是编程语言」         → [0.12, 0.87, 0.34, ...]
「Python 是一种写代码的工具」 → [0.13, 0.85, 0.36, ...]  ← 向量很接近
「今天天气不错」              → [0.78, 0.02, 0.91, ...]  ← 差很远
```

常用模型选择：

| 场景 | 推荐模型 |
|------|----------|
| 中文 / 多语言 | `paraphrase-multilingual-MiniLM-L12-v2` |
| 纯英文，速度优先 | `all-MiniLM-L6-v2` |
| 追求精度，预算够 | OpenAI `text-embedding-3-small` |

### 3.3 存入向量数据库

向量转好之后存进向量数据库，方便后续快速检索。代码示例用的是 **ChromaDB**，本地跑，不需要额外部署，适合学习和小项目。

## 四、第二阶段：查资料、作答

### 4.1 召回（Retrieval）

用户提问后，系统把问题也转成向量，然后去数据库里找最相似的几个片段。这一步叫召回，是粗选。

常见方式：

- **向量检索（语义搜索）**：找语义相似的内容，即使用词不同也能找到
- **BM25（关键词检索）**：传统的词频匹配，精确但不理解语义
- **混合检索**：两者结合，互补短板，效果通常更好

### 4.2 重排（Re-rank）

召回是粗选，用的是速度快但精度一般的向量相似度。重排是精选，用更精确（也更慢）的交叉编码器（Cross-Encoder）模型，把召回的候选结果重新打分排序。

```
向量检索（Bi-Encoder）：快，适合从海量数据里粗筛
重排（Cross-Encoder）：慢但准，适合对少量候选结果精排
```

数据量小、精度要求不高的场景，重排这步可以跳过。

### 4.3 生成（Generation）

把筛选出来的片段拼成「上下文」，连同用户问题一起交给 LLM：

```
提示词结构：
你是一个助手。请根据以下资料回答问题。

【资料】
{检索到的相关片段}

【问题】
{用户的问题}
```

LLM 基于这些真实资料作答，而不是靠自己「记忆」生成，幻觉大幅减少。

## 五、关于 Dify：不写代码也能搭 RAG

上面说的这套流程，如果全部自己用 Python 实现，从零开始要花不少时间。

我接下来准备用 **[Dify](https://dify.ai)** 来搭一套真正可用的 RAG 工作流。Dify 是一个低代码 AI 应用开发平台，把上面这些步骤都封装成了可视化的「积木块」，拖拖拽拽、配置一下就能把 RAG 跑起来。

刚开始接触 Dify 时我也没搞明白「在平台上开发智能体」是什么意思——感觉很玄乎。后来我的理解是这样的：

**没有 Dify**：你需要写代码，手动连接模型 API、配置向量数据库、编写检索逻辑、写提示词模板，每一步都要自己实现。

**有了 Dify**：这些底层的东西它都帮你打包好了。你在网页界面上：

1. 上传文档 → Dify 自动完成分片、向量化、存储
2. 拖一个「知识库检索」节点 → 搜索逻辑配好了
3. 拖一个「LLM」节点 → 接上模型
4. 连线、配提示词 → 逻辑串起来了
5. 点发布 → 直接拿到一个可用的 API 或聊天界面

本质上，Dify 做的是把「软件工程工作」转成「配置工作」。不需要特别精通编程，也能把一套完整的 RAG 流程跑起来。Dify 内部集成了 LangChain 等框架的能力，所以配出来的东西并不是玩具，是真正可用的。

## 六、各环节可以怎么优化

学完基础流程之后，有几个方向可以继续深挖：

| 环节 | 优化思路 |
|------|----------|
| 分片 | 按语义边界智能分片；父子分片（大块存储、小块检索） |
| 索引 | 混合索引（向量 + 关键词）；一个片段存多个向量 |
| 召回 | 多路召回；查询改写（把用户问题改写成更适合检索的形式） |
| 重排 | 数据量小可跳过；数据量大建议加上 |
| 生成 | 让 LLM 在答案里标注来源；支持多轮对话记忆 |

## 七、常见问题

**Q：分片大小怎么选？**

经验值是中文 100~200 字、英文 300~500 字符。原则是保证语义完整。如果不确定，先用默认值跑起来，再根据实际效果调整。

**Q：必须要重排吗？**

不一定。数据量小、精度要求不高可以跳过。如果发现召回结果里排名靠前的不够准确，再考虑加上。

**Q：Embedding 模型怎么选？**

中文或多语言场景用 `paraphrase-multilingual-MiniLM-L12-v2`，效果稳定，本地跑也行。追求更高精度且有预算，可以用 OpenAI 的接口。

**Q：怎么知道效果好不好？**

最简单的方法：准备几个你知道标准答案的测试问题，实际跑一遍看输出质量。更系统的做法是用 LLM 给回答打分（自动评估）。

## 八、配套代码

本文配套的完整 Python 实现见 [rag_system.ipynb](./rag_system.ipynb)，涵盖：

如果只是想理解流程，不需要配 API Key 也可以跑检索部分验证逻辑。

## 九、一句话总结

RAG 的核心思想：**不靠 AI 瞎猜，先找到真实资料，再让 AI 基于资料回答。**

这样既用上了 AI 的语言能力，又保证了答案准确、有据可查。
