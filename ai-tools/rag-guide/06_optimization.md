# 六、各环节可以怎么优化

> RAG 系统搭起来之后，效果不理想怎么办？各个环节都有优化空间。

---

## 分片优化

分片是对最终效果影响最大的环节之一，值得反复调试。

| 优化方向 | 具体做法 |
|---------|--------|
| 调整分片大小 | 中文一般 300-500 字，英文 512 token 左右，根据文档特点调整 |
| 增加重叠 | overlap 设为分片大小的 10%-20%，防止关键信息被切断 |
| 语义分片 | 按段落/章节边界切，而不是硬切 |
| 父子分片 | 存小块用于精确检索，检索命中后返回其父级大块给 LLM |
| 添加元数据 | 每个块附上来源文件名、页码、章节标题，方便过滤和引用 |

---

## 索引优化

| 优化方向 | 具体做法 |
|---------|--------|
| 选更好的 Embedding 模型 | 中文场景优先用 BAAI/bge-m3，效果明显优于通用模型 |
| 混合索引 | 向量检索 + 关键词检索（BM25）结合，覆盖语义和字面匹配 |
| 层级索引 | 先检索摘要定位章节，再在章节内精确检索 |

---

## 召回优化

| 优化方向 | 具体做法 |
|---------|--------|
| 调整召回数量 | top-k 不是越多越好，太多会引入噪音，一般 3-10 |
| 相似度阈值 | 设置最低分数线，过滤掉相关性太低的结果 |
| 查询扩展 | 用 LLM 把用户问题改写成多种表达，分别检索后合并 |
| HyDE | 先让 LLM 生成一个假设性答案，用答案去检索（比直接用问题检索效果更好）|

---

## 重排序优化

| 优化方向 | 具体做法 |
|---------|--------|
| 加入重排序 | 召回 20 个，重排后取前 3，精度提升明显 |
| 选合适的重排模型 | BAAI/bge-reranker-v2-m3 中英文效果好 |
| Cohere Rerank | 云端 API，效果强但有成本 |

---

## 生成优化

| 优化方向 | 具体做法 |
|---------|--------|
| Prompt 优化 | 明确角色、明确说「没有信息就说不知道」、要求引用来源 |
| 上下文压缩 | 召回内容太长时，先用小模型压缩再送给大模型 |
| 多路召回融合 | 多个知识库分别检索，结果融合后再生成 |
| 流式输出 | 提升用户体验，答案逐字显示而非等待全部生成完 |

---

## 评估体系

优化必须有指标支撑，否则不知道改了有没有效果。

| 指标 | 说明 |
|------|------|
| 召回率（Recall） | 相关文档有没有被检索到 |
| 精确率（Precision） | 检索到的文档有多少是真正相关的 |
| 答案相关性 | 生成的答案和问题是否相关 |
| 忠实度（Faithfulness） | 答案是否基于检索内容，没有编造 |

推荐用 [RAGAS](https://github.com/explodinggradients/ragas) 框架做自动化评估。
