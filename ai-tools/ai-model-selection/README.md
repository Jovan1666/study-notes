# 大模型怎么选？—— 一个真实用户的梯队排名

> 不看跑分，看实际干活的表现。这是我用了将近一年各种模型之后，给出的主观评价。

---

## 前言：模型为什么比工具更重要

[上一篇文章](./coding-tools-journey/)讲了 AI 编程工具的选择，这篇把目光往回拉一点——讲大模型本身。

AI 编程工具（Cursor、Claude Code、Codex 这些）本质上是大模型 + 工具调用能力的组合。大模型是大脑，决定了它能不能真正理解你的需求、写出靠谱的代码。工具再花哨，大脑跟不上，一切白搭。

所以选模型这件事，比选工具更本质。

---

## 评价标准说明

我没有用固定测试任务跑分，全部基于实际工作中的使用感受打分。

评价维度：
- **代码能力**：一次成型率、逻辑深度、后端 vs 前端表现差异
- **文本处理**：写文档的质量、篇幅是否充实
- **图像理解**：看图识图的能力
- **推理能力**：复杂问题的拆解和分析
- **使用体验**：速度、稳定性、限流情况

---

## 梯队划分一览

| 等级 | 模型 |
|------|------|
| **夯**（最强） | Claude Opus 4.6 / 4.5，GPT-5.4 |
| **顶级** | Gemini 3.0 Pro，Claude Sonnet 4.6，Codex 5.3 |
| **人上人** | GLM-5，Kimi K2.5 |
| **NPC** | DeepSeek，MiMO-2-Pro（小米） |
| **拉完了** | MiniMax M2.5 / M2.7 |

---

## 夯（最强梯队）

### Claude Opus 4.6 / Opus 4.5

当之无愧的最强模型。文本处理、图像理解、代码编写，每一项都是顶级，没得说。

代码方面，一次成型的准确率极高，多文件业务逻辑处理能力远超其他模型。图像理解也是强项，我经常用 `@Read` 工具让模型看代码截图，它能准确描述图中的代码结构和逻辑。写文档的能力同样扎实，输出的内容有深度、有结构。

**但是有一个致命问题：国内使用极其困难。**

Anthropic 对国内的访问风控非常严格，官方渠道充值有封号风险。中转站是国内用户的主要选择，但水很深：不少商家会在一个大号池里掺杂 opus4.6 和其他模型，用模型映射糊弄用户，实际体验和真正的 opus 差距很大。而且中转站经常不稳定——输出到一半卡住、限速、限流是常态。

再加上不同编程 IDE 的系统提示词会干扰模型输出，综合来看体验打了折扣。

但即使有这些干扰，opus 依然领先其他模型一个身位。只要你有一个稳定可用的渠道，opus 仍然是编程的首选。

Opus 4.5 是弱化版的 4.6，能力依然很强，但其他模型已经很难追上了。

### GPT-5.4

GPT-5.4 是 Claude 国内风控浪潮期间的「最佳替代」。

和 Opus 相比，GPT 的语义理解能力更强——它更能捕捉我模糊表达背后的真实意图，减少了对提示词优化的依赖。这一点在日常工作中很实用，有时候懒得写得很详细，GPT 反而能猜到我想干嘛。

早期我不太愿意用 GPT 的一个核心原因是：**太慢了**。同样一个问题，Opus 可能 2 分钟跑完，Codex（GPT 的编程模型）要跑 10 分钟以上。而且 Codex 5.3 的文本能力很一般——分析到位了，写出来的东西却惜字如金，文档没眼看。

GPT-5.4 解决了这两个问题。运行速度大幅提升，最明显的问题得到了很好的改善。文本能力也跃升了一个档次，我用它写的项目文档已经能达到「不让我觉得不行」的水平——对于 GPT 来说，这已经是很大的进步了。

推理能力强，适合做代码测试、定位问题。后端代码能力也没问题。唯一的问题是前端审美不行，生成的界面比较朴素。

目前我自己的号池里用着 GPT-5.4，作为主力模型的备选和配合工具。

---

## 顶级（第二梯队）

### Claude Sonnet 4.6

Sonnet 是 Opus 的「轻量版」，但放在第二梯队里依然是强力选手。

日常的文本处理、图像理解、代码编写都能胜任，代码能力比 Opus 弱一些，但对于大部分日常开发任务完全够用。胜在性价比高，速度也比 Opus 快不少。

如果你没有稳定渠道用 Opus，或者任务难度没那么极端，Sonnet 是一个很务实的选择。

### Codex 5.3

Codex 是 OpenAI 的编程专用模型，也就是 GPT 系列的代码版本。

推理能力很强，适合用来做代码测试、定位后端问题。后端代码质量没话说。

缺点是慢，而且文本写作上惜字如金，不适合用来写文档——这大概也是 OpenAI 后来把 Codex 能力整合进 GPT-5.4 的原因。

### Gemini 3.0 Pro

Gemini 是一个「偏科选手」。

**强项：前端设计 + 图像处理。** Gemini 有很好的审美，搭配 Figma 或者 Stitch 这类前端制作工具，能一下子生成让人眼前一亮的前端 Demo。这是我在其他模型上很少见到的能力。

**弱项：后端 + 稳定性。** 如果不用 Stitch 这样的工具辅助微调，而是直接通过对话让模型修改，它有时候会「跑偏」——可能是因为幻觉，也可能是有自己的想法，导致结果和预期差很多。Gemini 的网页端幻觉问题比较明显。

写后端代码对比同期的 Claude 和 Codex，弱了不少。但简单的后端逻辑还是能写的。

背靠 Google 这样的厂商，Gemini 的生态和服务稳定性有保障，而且它的多模态能力在生图一致性测试中表现最稳定。

## 人上人（第三梯队：国产模型）

### GLM-5

GLM-5 是我认为国产模型里最强的。

文本编写和项目文档生成能力让我很惊喜——当时用的时候没有抱太高期望，但实际效果和 GPT-5.4 写出来的感觉差不多，甚至让我觉得有点惊艳。

代码能力也不错。我给 GLM 一个任务描述，它能比较准确地理解意图并执行。换到其他国产模型（比如 MiniMax），我详细阐述任务它还会出错，GLM 基本上我大概说一下它就懂了。

**最大的痛点是算力和限流。** 智谱的算力不够，我用 GLM-5.1 的时候几乎一直在被限速，体验很差。流量一大就开始卡，等流量稳定了我再深入用。

GLM-5.1 出了但还没正式体验，限流问题如果能解决，GLM 绝对能往上走一档。

### Kimi K2.5

Kimi 给我留下的印象主要是前端设计能力——这是我刚用 Kimi 时的核心体验，搭配图像理解能力，处理多模态任务时表现不错。

最近又深度用了一周，发现图像理解能力确实比较强，更多方面的表现还需要继续探索。

和 GLM 一样，Kimi 也是国产模型里值得关注的选手，在追赶国际主流模型的路上走得挺快。

---

## NPC（第四梯队）

### DeepSeek

DeepSeek 在我早期使用 AI 时是主力选手——那时候问问题用它，推理能力确实强。

但它的局限也很明显：之前的版本缺乏多模态能力，不支持图片输入，只能处理纯文本。对于现在的应用场景来说，功能显得单一了。

DeepSeek V4 迟迟不出，V3.2 版本的多模态支持也比较弱，目前在我的工具箱里优先级不高。

### MiMO-2-Pro（小米）

小米自研的模型，刚发布的时候简单体验了一下。

给我的感觉很像 MiniMax——体验不算好，具体能力没有深入测试，不敢乱评价。等后续版本再看。

---

## 拉完了（末尾梯队）

### MiniMax M2.5 / M2.7

MiniMax 最大的优点就是**便宜、量大**。

然后就没有然后了。

我之前用 M2.5 养「龙虾」（一个需要模型持续交互的任务），笨得不行。让它看图片经常看不了，以为发了图片它应该能识别，结果经常报错。M2.7 出了之后我以为会有改善，实际用下来和 M2.5 几乎没有区别。

推理能力不行，文字处理不行，图像识别不行——除了便宜，我在这个模型上找不到任何让我想继续用的理由。

## 按需选择建议

| 需求 | 推荐 |
|------|------|
| 极致编程能力 | Claude Opus 4.6（稳定渠道的前提下） |
| 深度调研 + 方案设计 | GPT-5.4 |
| 前端设计 + 生图一致性 | Gemini 3.0 Pro |
| 国产平替（文档能力强） | GLM-5（等算力稳定） |
| 国产平替（多模态） | Kimi K2.5 |
| 预算有限，随便用用 | MiniMax（便宜但别抱期望） |

---

## 结语

每个模型发布时都会给出一堆 benchmark 跑分，看着各个都逼近最强。但实际用起来完全是另一回事。

benchmark 是纸面数据，长期工作中模型能否保持稳定水准，才是真正的能力体现。

以上评价完全基于我的主观使用感受，每个人用同一模型的感觉可能不同。如果有不同看法，欢迎交流。

---
