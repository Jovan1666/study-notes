# 一、早期探索：IDE 工具

> Cursor 初遇、通义灵码、Windsurf、Kiro、Warp

---

## Cursor —— 打开新世界的那扇门

最早接触 AI 辅助编程是 25 年 7 月，实习期间遇到了一个靠自己手写代码和当时的知识储备根本完不成的任务。那会儿我对 AI 的使用还停留在把问题贴给网页版 DeepSeek，让它帮我回答——但这个方式实在太慢，而且 DeepSeek 写代码的能力也没那么强。

后来上网求助，看到有人在分享 Cursor 有多厉害，就去 B 站学怎么下载和配置。那时候 Cursor 还有无限续杯，29.9 包月。现在回想起来，那个时候 AI 还没有以这种爆发式的形态呈现在大众面前，各种薅官方羊毛的途径还很多，现在已经基本不可能了。

配置好之后，我用了一个下午，完成了原本计划要花一周的工作。当时最强的写代码模型好像还是 sonnet 3.5，换成现在的模型，那点工作可能三四分钟就写完了。但对当时的我来说真的很震撼——我一直在想，要是网页版的 AI 能直接操控我的项目就好了，因为网页版只能看到我复制给它的片段。而 Cursor 实现了这件事，第一次体验的感觉确实很冲击。

---

## 通义灵码 —— 免费插件里的老实人

用完 Cursor 之后，因为要做 Java 作业，就在 VSCode 里装了阿里的通义灵码插件。免费的插件，用来应付学校作业完全够用，作业的难度也没有超出它的能力范围，完成度很高。

对于现在的实际开发环境来说可能确实不够用了，但对一个免费插件这么高要求也没什么意思。该给的认可还是要给。

---

## Windsurf —— 用得最频繁，但体验最难受

差不多是 25 年 11 月开始深度使用，在我进入现在这种固定工作模式之前，Windsurf 可能是我用得最多的编程 IDE。

先说一个算优点也算缺点的东西：它每次工作完之后都喜欢自动写一堆文档。出发点可以理解，上下文窗口短，多备一些文档能让 AI 了解当前的项目状态。但实际用下来，写出来的文档量太大、太杂，没用的文档满天飞，还和真正有用的文档混在一起，对开发者反而是干扰。

上下文短是它的核心缺点。那时候我还不懂上下文管理这回事，一个对话框一直用下去，越到后面 AI 越搞不清楚状况，新任务来了把之前的全忘了，当时一直觉得奇怪怎么老是忘，后来才明白是这个原因。

另一个让我受不了的问题是终端命令跑不通。写完测试代码让它自己跑，程序一启动就卡死，不是我在终端手动输，而是让程序自己执行——必卡。重装了好几次，还是这样，到现在也不确定是不是只有我有这个问题。

用的模型好像是 opus 4.1 和 sonnet 4.5 那个时候，具体记不太清了。问题描述得已经很详细了，但多次尝试还是解决不了。

最后是在完成了一个比赛项目之后退坑的。体验确实不舒服。

---

## Kiro —— 在闲鱼捡到的，学到了一个新概念

Kiro 是在闲鱼偶然发现的，有人在卖无限续杯，就去体验了一下。那时候好像已经有 opus 4.5 了，就在 Kiro 上用这个模型。

工作不乱写文档了，反而是在 Kiro 里能按要求写出很详细的大型文档，这成了它的一个优点。内容质量先不谈，光是能写那么长，当时就觉得还不错。

但整体用下来感觉有点笨，效果没有 Windsurf 顺手。可能也有一部分原因是用 Windsurf 用久了，突然切换过来一时没适应。

它有一个对我来说很有价值的功能：上下文满了会自动压缩，开一个新的对话窗口继续。正是因为这个，我第一次意识到上下文管理是怎么回事——原来 Windsurf 一个对话框用到底、越来越「失忆」，根源就在这里。这个认知算是 Kiro 给我上的一课。

还有 chat 和 spec 两种模式，简单说就是一个用来聊产品，一个用来构建工程框架并执行。两种模式的区分让用户清楚知道自己现在处于哪个阶段，这个设计挺好的。

没有深入使用，体验不算丰富，但 Kiro 的故事还没结束——后面我用它来反代 opus 4.6 模型，这是后话。

---

## Warp —— 理念超前，但生态没跟上

Warp 和 Kiro 的使用时间有重叠，大概是 26 年初开始用，之后重回 Cursor 也和这段时间有交叉。

第一次体验非常爽。它的布局和传统 VSCode 风格的 IDE 不一样，更偏命令行的工作方式，呈现给用户的过程都简化了，更多是结果输出——这个理念我觉得很符合 AI 开发的发展方向：过程交给 AI，人负责搭建、约束、最终审查输出的结果。

和我现在用 Codex 桌面端的感受很像。最大的不同点是权限——Warp 可以赋予 AI 很高的权限，布置好任务之后全部交给它，中间不需要手动确认任何步骤，一直到工作完成才停下来。让它跑测试的时候特别爽，测试代码和任务布置下去，等结果就行了。

后来回过神来，这其实是挺危险的行为。不过有意思的是，即使是开发者群体，对于给 AI 赋权这件事的边界也比较模糊——AI 想访问什么，基本都会允许。现在有沙盒机制来保护数据安全，但边界的问题其实还是没有完全解决。

让我最终离开 Warp 的原因是 MCP 和 skill 的配置体验太差。那时候 MCP 和 skill 已经开始盛行，但很多常用的 MCP 没有提供 Warp 的配置文件，配置起来很麻烦；skill 功能当时好像直接不支持配置。相比之下 Cursor 在这方面做得好很多，很多插件可以一键配置。

另外对不熟悉命令行的用户来说，Warp 的上手门槛比 VSCode 系的 IDE 高，很多在别的工具里鼠标点一下就能做到的事，Warp 需要手动输指令。

后来 Warp 好像做了比较大的更新，界面变了不少，但我之后没有再深度体验了。
