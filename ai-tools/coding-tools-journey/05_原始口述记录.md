# 原始口述记录

> 以下是作者在整理本文时的原始口述内容，未经润色，保留最真实的第一手体验感受。

---

我最先使用的这种辅助编程的其实是cursor，那是在25年7月份，实习期间有一个任务，当时手写代码和知识储备都不足以支撑完成那个任务，当时对ai的使用还停留在把问题交给网页端的deepseek，让deepseek回答我，但是这样非常慢而且deepseek写代码并没有那么强，然后我上网求助，看到别人在分享说cursor多厉害厉害，然后我就去b站学习，怎么下载和使用，那个时候还有cursor的无限续杯，29.9的包月，现在回想起来那个时候ai还没有以这种爆发示的形态呈现在大众面前，还有很多途径薅官方羊毛，现在是不可能了，然后我照着网上这么配置好之后，使用这个cursor，只用了一个下午，完成了我当时计划要1周的工作，当时最强的写代码模型好像还是sonnet3.5，别提现在的模型，对当时的工作可能三四分钟就写完了，当时就很震惊，因为对于一个项目，我当时想的就是，要是这个网页端的ai能直接操控我的项目就好了，因为网页端的ai看到我复制给他的，然后cursor实现了，当时对于刚体验的我真的非常震惊。

然后之后我就是使用了vscode里面的通义灵码的插件，当时是要做java的作业，使用这个插件，对于免费的这个插件，他还是很合格的，而且学校作业的难度并不会超出这个模型的能力，所有完成度也很高，不过对于我现在的实际发开发环境可能有点不够用了，但是对于这个插件，我还是给予很高的认可，免费的还那么高要求干嘛？

再往后就体验了windsurf，这个windsurf可能是我在现在固定工作场景之前，使用非常频繁的编程ide，先说优点，这个有点其实是一个缺点，但是对于我想说的缺点这还缺失算一个优点，差不多是在25年的11月份我深度使用，这个windsurf总喜欢再一次工作之后写一堆文档，这个就是他的优点，可能是要解决上下文小的问题，多准备这些文档，能让ai知道现在的实际工作情况，但其实写那么多工作文档很糟糕，一堆没用的文档浪费空间，还跟有用的文档混在一起干扰程序员，缺点就是上下文太短，我也不知道这个问题现在解决没有，另一个就是可能是我当时使用的模型还没那么强，好像是opus4.1和sonnet4.5那个时候，具体是不是记不清了，就是我已经在很详细的阐述我的问题了，他多次解决下来解决不了，还有就是我的windsurf跑不通我的终端命令，就是一跑就卡在那里走不下去，我不知道是不是只有我有这个问题，反正重装好几次也是这样，写个测试代码，需要终端去跑的时候，就是让程序自己去跑，不是我自己在终端输入，程序一跑就卡住，就在这样的开发环境下，我完成我的比赛项目之后就退坑了，体验确实不舒服。

下一个就是kiro，这是我在海鲜市场偶然发现的，有人在卖无限续杯，然后我就去体验了一下 ，怎么说呢那个时候好像已经有opus4.5，我在kiro上使用这个模型，工作确实不乱写文档了，反而在kiro里面写详细的文档符合我要求的文档成了他的有点，他能写很长的文档，内容质量这一方面怎么说先不谈，就先说能写那么多文档，当时觉得还挺不错，但是对于这个编程ide我并没有深入使用过，因为我当时使用下来，觉得他笨笨的就是给我呈现的效果并没有在windsurf的感觉好，可能是我当时长期使用windsurf突然的转变让我对他并没有完全接受，但是他有一个很好的就是当上下文满了，他会自动压缩，换一个窗口，我当时使用windsurf的时候，对于上下文容量这个还不懂，然后就是用windsurf一个对话框能用非常久，然后他也不自动压缩，就这样后面新任务来了，就把前面的忘了，怪不得当时老忘了我要干嘛，kiro还有就是chat和spec这两个区分，我说的简单一点就是，一个是聊产品，一个是构建产品的工程框架然后执行。因为我使用的不是很深入，所以给出的结论可能不是很片面，缺点就是模型有点笨笨的，即使使用的是顶级模型，优点是写支持写一个很大的文档，chat和spec的两种交互模型让用户知道自己现在处于什么样的一个阶段，自动压缩上下文并且开启新的对话在当时也是让我学到了新知识，我的使用体验并不是很丰富，但是和kiro的故事还没有结束，后面我还会再提。

就是我这些工具并不完全是时间顺序上这个用完然后用下一个，这些基本是有时间上的重合的，一个时间段我可能在探索多个编程ide，就比如我下一个要讲的warp，其实这个和kiro是由重合时间的，kiro和windsurf是有重合时间的，warp讲完之后我又回归到cursor这也是有重合时间的。

接下来讲warp，这个是初次体验非常爽的一个编程ide，他的布局方式也不同于传统的vscode这样的编程ide，应该是在26年初这个时间段开始使用，他给我的感受，和我现在使用codex桌面端的感受很像，直接讲不同点，在那个时间段，warp不同于其他的编程ide就是他能赋予很高的权限，就是其他的编程ide执行任务的时候老来问你可不可以，赋不赋予权限这些，要你手动确认执行下一步，但是warp可以赋予很高权限之后，全部交给ai监管，我也是后知后觉这个其实是很危险的行为，但是当时真的很遍历，尤其是让他帮我跑测试的时候，真的很爽，测试代码还有任务这些布置下去，知道之后全部解决并且测试跑通他才停下来，中间也不让你确认什么，我体验下来不舒服的点就是，当时mcp和skill已经开始盛行了，但是很多实用的mcp，warp配置起来不便利，有些甚至没有提供给warp的mcp配置文件，相比较这个cursor就比较强，很多外部插件cursor都可以一键配置，没有那么多复杂的自己配置情况，还有一个就是skills，我当时使用的时候skill开始兴起，但是当时warp好像还配置不了，就是因为这个我慢慢寻找可以配置skill的编程ide了，之后就又回到了cursor，总结一下warp，缺点就是感觉他总不能在一个新技术出来的时候立即适配，用户想用但是没法用确实很烦，还有就是权限的问题，但是我感觉即使是开发人员，对于赋予ai权限这个问题，他们也没有很好的边界，虽然现在有了沙盒来保护数据安全，但是从openclaw这么火能看出来，即使是开发人员，对于权限这件事情的边界也是比较模糊的，ai想要访问的话基本都会允许，优点也不少，比如赋予权限之后一直工作直到工作完成，现在可能大家都能实现了，但是在当时确实让我用着很舒服啊，warp的实用，虽然是编程ide但是和其他的不同，他更走的是类似codex或者cc命令行这样的工作模式，就是呈现给用户的过程都简化了，更多是结果的输出，让用户更注重结果，我感觉这还是挺符合ai开发的发展趋势的，过程交给ai，人负责搭建，约束，最终审查ai输出的结果，现在warp好像做了不小的更新，但是我后面也没实用体验了，只感觉操作界面不一样了，哦对了，还有个就是，如果命令行这种工作模式不擅长的，用warp可能不像使用那种vscode的编程ide那样方便，因为warp有不少别的编程ide只需要鼠标点击的功能，warp需要输入指令。

后面我们来讲cursor，我们又回到了cursor，之前那次使用很粗糙和浅显，知识打开了我新世界的大门，这次我的使用很深入，差不多使用了7000刀左右，是的额度7000刀，换算下来可能有几百亿几千亿的tokens，cursor是我觉得生态最好的编程ide，就是如果是初入这个领域的小白，使用cursor的话也能很快上手，cursor最直观的就是它分为四个模式，ask，agent，plan，debug，这四个名字就是"人如其名"，ask就是和ai交互，聊产品这些，agent就是智能体调用工具还有准备好的开发文档去开发，plan就是写开发文档，制定开发文档这些，debug就是查bug，在可能出现bug区域加入测试代码，一点点去确定问题所在,这些模式的主要功能就是这些，我不详细说，还有就是一个问题可以同时打开很多模型来回答，自己选择最好的那一个，但是我当时不太接受的了，我感觉ai输出的信息已经很爆炸了， 同时再接受那么多相似的信息，然后去对比其中谁更好，感觉太累了，我都是对应工作使用对应模型，设计和制作前端用gemini，快速搭建后端代码用claude，测试，改bug和针对性修改前后端用gpt，这样子发挥了各个模型的强处，还只需要处理一方输出的信息，我的个人感受，在编程ide里面，cursor像是手机里的苹果，cursor配置skill，mcp这些都很变量，不少mcp可以一键配置，cursor也有专门管理skill和创建skill的地方，优点很多，cursor绝对是我初入大模型开发这个领域的白月光没得喷。

简单聊一下codebuddy，腾讯出品，简单使用过，我不太了解所以不评价，国产还是希望能越做越好。还有就是antigravity，这个我初次使用确实不错，使用自己的谷歌账号，ip要符合要求，能免费使用不少额度，claude的额度也不少，谷歌自研的gemini额度很多，但是因为账号ip的问题，很多想用的人被ip限制根本用不了，还有就是反代的问题，antigravity提供的额度一降再降，即使是付费用户也基本用不了多久就被限额，antigravity这个编程ide我并没有深入体验，但是和他的故事还没结束。

插件这个方面用过augment，我没怎么深度使用过，网上评价好像还挺好的，但是我使用下来就是感觉任务执行太慢，消费太高，一点点问题就用了不少钱了，用不起，还有个就是cline（已经改名），这个是使用apikey配置，调用大模型api然后使用，这个是开启了我的下一个篇章。我上面说的这些编程ide，不算插件，如果不是公司统一配置或者资金充足的情况下，个人开发者已经不适合使用了，价格昂贵，额度很少，国内各种破解薅羊毛的方式相继被风控，于是走向了我的下一个阶段。

我是在网上刷到claudecode很厉害，应该是25年11月份左右开始接触，然后我就去海鲜市场找代充180块充了一个claude的会员，初次体验是命令行的工作模式，其实很不习惯， 但是现在已经很接受了，我当时是网页端使用，用它写文档，当时并不懂该怎么用，后来26年初，opencode的爆火，然后跟claudecode对比，我就开始学习这两个编程工具这两个agent怎么用，我先说我的结论，claudecode肯定是比opencode强的，而且opencode一直有个问题就是用久了就会突然批量输出一些符号还停不下来这样的bug，但是opencode是开源的，而且对国内用户优化，cc就不提了，对国内用户其实很不友好，风控很严重。opencode有一个很厉害的插件就是omo，oh-my-opencode，就是一种多智能体协作的方式，一个强力的模型作为大脑，然后分配给模型去做擅长的工作，这个挺厉害的，而且opencode对用户都会提供不少免费的额度，最新才出的mimo-v2-pro模型也提供免费的额度，opencode还是很支持小白先上手体验的。cc虽然对国内用户不友好，但是架不住agent能力确实很强，而且模型也很强，工作起来确实厉害，我并不是崇洋媚外，而是基于我实际的使用来分享使用体验。cc的具体使用细节这里不讲，就说几个点，一个是当时让我很喜欢的功能就是，agent team，有点像omo的功能，但是在cc里面使用opus4.6这个世界顶级模型，所有任务都交给他来做，一个大脑agent去创建新agent然后分配工作，大大提高效率，但是相应的token的消耗也在不止不觉间消耗的很多。还有推出fast模式，六倍消耗但是效率更高，因为模型太贵所有我不太敢用，我只记得当时开了这个模式，一句话花了我六块钱，现在cc确实很强cowork我还没使用过，但是我看过很多技术博主的视频也很强，感觉cc看到，不管是普通用户还是技术人员对于openclaw都能赋予这么高的权限开始放飞了，最近才出的产品也就是和龙虾一样，甚至没有那么麻烦的配置问题，因为这个cc每次不管做什么都要访问我的意见让我开始下一步。具体的使用可以去学习相关博主的视频，这里就不多讲。

因为国外风控的问题 ，cc想要在国内流畅性价比的使用已经很难了，加上奥特曼也就是openai吸引用户给除了那么多优惠政策，我开始使用codex，其实我之前也用，但是模型速度实在是慢，做一些需要强推理的任务才使用，但是gpt6.4出了之后，还提供fast模式，我开始使用，我觉得gpt5.4对比之前确实有不少提升，然后再搭配上codex的桌面端应用windows系统也上线了，就深入使用了一下，先说优点，写文本的能力确实比以前强了不少，至少codex5.3和gpt5.4的文本处理我觉得不是一个档次，然后使用这个codex app扔个大型任务给他，然后基于权限就能一直工作，知道任务完成，虽然我们都知道，这种情况任务不会全部的完成，但是这个工作不需要我去确认下一步继续还是挺不错了，这个就和warp很像，所有我很喜欢准备好测试代码，然后扔给codex去测试，不需要我干预，他能找出问题解决问题。openai的模型除了慢了点，但是作为模型的开山鼻祖，能力还是可以的，最近codex又更新了新内容，我还没来的及体验，具体的使用教程网上有很多，这里不细讲。

我现在的工作模式就是vscode+cc插件，然后使用cc-switch给cc配置各种各样的模型，opus4.6还有很多国产的模型，基本就是codex和claudecode搭配使用，我不怎么用gemini，我感觉cc搭配skill写出来的界面不比gemini差，主要还是觉得gemini降智太快了，很快就不懂我在说什么了。我这期主要是聊ai编程工具，对于模型可以下一期好好聊聊模型的选择。上面说关于kiro和antigravity的故事没结束，其实是我后面有段时间使用的opus4.6模型就是通过kiro和antigravity反代出来的，不过现在风控已经严重了，使用体验很不好又卡又慢。
